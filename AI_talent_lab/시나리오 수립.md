### 핵심 사용자 시나리오

본 서비스는 취합 담당자가 메일을 시스템에 직접 등록하는 방식이 아니라, Gmail에 도착한 이벤트를 Agent가 인지하고 필요한 후속 행동을 이어서 수행하는 흐름을 기준으로 한다. 사용자는 자동 확인을 켜두거나 화면의 `지금 실행`을 눌러 동일한 처리를 시작할 수 있다.

### 시나리오 1: 수신함 자동 확인 및 취합 요청 처리

* ID: SC-001

* 상황:
  취합 담당자의 Gmail에 일반 메일, 취합 요청, 스팸이 섞여 도착한다. 취합 요청에는 첨부 양식이 있을 수도 있고, 자연어 요구사항만 있을 수도 있다.

* 목표:
  Agent가 메일을 분류하고 취합 요청의 요구사항·양식·수신자·위험도를 판단한 뒤, 자동발송 또는 사람 승인까지 준비한다.

* 사전 조건:

  * 백엔드가 실행 중이고 Gmail 읽기 인증이 완료되어 있어야 한다.
  * UI에 자동 확인 여부, 실행 시각, 타임존이 저장되어 있어야 한다.
  * 실제 자동발송 시 허용 도메인과 Gmail 보내기 인증이 설정되어 있어야 한다.

* 상세 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 담당 기술/Agent |
| --- | --- | --- | --- |
| 1 | 자동 확인을 켜거나 `지금 실행`을 누른다. | 최근 새 메일과 첨부를 수집하고 message-id 중복을 제거한다. | APScheduler, Gmail API, Inbox Intake Agent |
| 2 | 별도 행동 없음 | 일반·취합·스팸으로 분류하고 취합은 요청·제출·질문·수정·연장으로 세분화한다. | Azure OpenAI Structured Output |
| 3 | 별도 행동 없음 | Supervisor가 분류, 신뢰도, 위험 플래그, 이전 observation을 보고 다음 Worker를 선택한다. | LangGraph Supervisor Agent |
| 4 | 별도 행동 없음 | 목적, 작성 항목, 마감, 대상, 주의사항을 구조화한다. | Requirement Analysis Agent |
| 5 | 별도 행동 없음 | 첨부 Excel이 있으면 재사용하고, 없으면 요구사항으로 새 양식을 만든다. | Template Design Agent, openpyxl |
| 6 | 필요 시 추가 수신자를 입력한다. | 명시 대상자를 우선 사용한다. 대상자가 없으면 원본 발신자+참조자를 기본 수신자로 정한다. | Directory Tool, Recipient Policy |
| 7 | 승인 대상만 검토한다. | 가이드와 요청 메일을 작성하고 Policy Gate 결과에 따라 자동발송 또는 승인 대기로 전환한다. | Communication Agent, Gmail API, Policy Gate |

* 입력 예시

```text
제목: 7월 시스템 개선 요청사항 취합
본문: 각 팀의 요청 시스템, 개선 내용, 긴급도, 사유를 7월 24일 17시까지 회신해 주세요.
발신자: requester@company.com
참조: planning@company.com
첨부: 없음
```

* 기대 출력

```text
상위 분류: 취합 업무
세부 의도: 신규 취합 요청
추출 항목: 요청 시스템, 개선 내용, 긴급도, 사유
마감: 2026-07-24 17:00
양식 결정: 신규 Excel 생성
기본 수신자: requester@company.com, planning@company.com
처리 결정: 신규 양식 생성 건이므로 승인 대기
산출물: 작성 양식, 작성 가이드, 요청 메일 초안, Collection Job ID
```

* 대체 흐름 및 예외

  * 첨부된 정식 양식을 그대로 사용하고 모든 안전 조건을 만족하면 자동 완료 또는 자동발송이 가능하다.
  * 신규 양식을 생성한 경우에는 사람이 양식과 메일을 확인한 뒤 발송한다.
  * 수신자가 없거나 허용 도메인 밖이면 자동발송하지 않는다.
  * 프롬프트 인젝션·피싱 위험이 있으면 Security Agent가 격리한다.
  * 요구사항이나 마감이 불명확하면 승인 대기로 보낸다.

* 성공 기준

  * 일반·취합·스팸과 취합 세부 의도가 구조화된 값으로 저장된다.
  * 첨부 유무에 따라 기존 양식 재사용과 신규 양식 생성이 정확히 분기된다.
  * 수신자 근거와 자동발송 차단 사유가 Action Log에 남는다.
  * 고정 24건 평가셋에서 Azure LLM의 상위 분류+세부 의도 일치율 100%를 재현한다.

### 시나리오 2: 작성자의 질문 자동응답 및 정책 변경 승인

* ID: SC-002

* 상황:
  취합 요청을 받은 작성자가 같은 Gmail 대화에서 작성 방법, 허용값, 마감 등에 대해 질문한다.

* 목표:
  원래 취합 Job의 사실만 사용해 단순 질문에는 자동으로 답하고, 기한·양식·예외 정책을 바꾸는 요청은 사람에게 넘긴다.

* 사전 조건:

  * 취합 요청 메일에 `[SC-...]` Job ID가 포함되어 있거나 Gmail thread가 저장된 Job과 연결되어야 한다.
  * 질문자가 해당 Job의 수신자여야 한다.

* 상세 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 담당 기술/Agent |
| --- | --- | --- | --- |
| 1 | 작성자가 요청 메일에 질문으로 회신한다. | 새 메일을 `collection/question`으로 분류한다. | Inbox Intake Agent, Azure OpenAI |
| 2 | 별도 행동 없음 | thread 또는 Job ID로 원래 취합 건을 찾는다. | Job Store, SQLite |
| 3 | 별도 행동 없음 | 마감, 작성 항목, 검증 규칙, 가이드에서 관련 사실을 검색한다. | Q&A RAG Agent |
| 4 | 별도 행동 없음 | 답변 초안을 만들고 사용한 근거 키를 기록한다. | Azure OpenAI, Structured Output |
| 5 | 승인 대상만 검토한다. | 단순 질문은 같은 Gmail thread로 답장하고, 정책 변경 질문은 승인 대기로 전환한다. | Communication Agent, Policy Gate, Gmail API |

* 입력 예시

```text
제목: Re: [SC-20260720-001] 7월 시스템 개선 요청사항 취합
질문: 긴급도에는 어떤 값을 입력하면 되나요?
```

* 기대 출력

```text
연결 Job: SC-20260720-001
근거: validation_rule.code_rules.긴급도 = [상, 중, 하]
답변: 긴급도는 상, 중, 하 중 하나를 입력해 주세요.
처리: 안전 조건 충족 시 동일 Gmail thread 자동 회신
```

* 대체 흐름 및 예외

  * “마감을 내일까지 연장해 주세요”, “다른 양식으로 제출해도 되나요?”는 정책 변경이므로 사람 승인으로 보낸다.
  * 질문자가 취합 대상자가 아니면 자동응답하지 않는다.
  * Job을 하나로 확정할 수 없거나 근거가 부족하면 답변을 추측하지 않는다.

* 성공 기준

  * 질문이 원래 취합 건과 정확히 연결된다.
  * 답변에 사용한 근거가 저장된다.
  * 정책 변경·비대상자·근거 부족 질문은 자동발송되지 않는다.

### 시나리오 3: 제출 검증, 자동 반려, 재제출 및 리마인드

* ID: SC-003

* 상황:
  여러 작성자가 Excel을 회신한다. 일부 파일은 정상이고 일부는 필수값 누락, 날짜·코드값 오류가 있다. 마감이 가까워져도 제출하지 않은 사람이 있다.

* 목표:
  회신을 자동 검증하고, 안전하게 고칠 수 있는 값만 교정하며, 오류가 남으면 사실 기반 재작성 요청을 보내고, 미제출자에게 리마인드한다.

* 사전 조건:

  * 취합 Job과 수신자 목록, 마감, 검증 계약이 저장되어 있어야 한다.
  * 제출 메일이 thread 또는 Job ID로 원래 취합 건과 연결되어야 한다.

* 상세 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 담당 기술/Agent |
| --- | --- | --- | --- |
| 1 | 작성자가 Excel을 첨부해 회신한다. | 제출 또는 수정본으로 분류하고 취합 Job을 찾는다. | Intake/Supervisor Agent |
| 2 | 별도 행동 없음 | 필수값, 날짜, 숫자, 코드값, 중복을 검증한다. | Validation Agent, pandas/openpyxl |
| 3 | 별도 행동 없음 | 날짜·코드값의 안전한 교정 후보를 만들고 코드 게이트로 확인한다. | Self-Correction Agent |
| 4 | 별도 행동 없음 | 재검증 결과 오류가 실제 감소한 경우에만 별도 교정본을 채택한다. | Deterministic Revalidation |
| 5 | 별도 행동 없음 | 오류가 남으면 행·컬럼·사유를 포함한 재작성 요청을 작성한다. | Communication Agent, Azure OpenAI |
| 6 | 별도 행동 없음 | 정상 제출은 완료자로 등록하고 전원 완료 시 최종본을 병합한다. | Submission Store, Merge Agent |
| 7 | 승인 대상만 검토한다. | 마감 임박 시 미제출자를 계산해 리마인드 초안 또는 정책 통과 메일을 발송한다. | Deadline Agent, APScheduler |

* 입력 예시

```text
작업번호: SC-20260720-001
제출자: user1@company.com
첨부: 생산팀_개선요청.xlsx
오류: 5행 긴급도='매우 급함', 8행 요청 시스템 누락
```

* 기대 출력

```text
검증 결과: 오류 2건
Self-Correction: 허용값을 추측할 수 없어 미채택
상태: 재제출 요청
반려 메일: 파일명, 행, 컬럼, 입력값, 허용 기준 포함
제출 현황: user1@company.com = 오류/재제출 대기
```

* 대체 흐름 및 예외

  * `2026.07.20`처럼 의미가 보존되는 날짜 정규화는 재검증 후 채택할 수 있다.
  * 필수값 누락이나 서술형 업무 내용은 Agent가 임의로 채우지 않는다.
  * 작업번호·thread를 찾지 못하거나 첨부가 없거나 파일이 손상되면 Supervisor가 observation을 받아 사람 확인으로 재계획한다.
  * 일시적 Worker 오류는 1회 재시도하고 재실패하면 사람 확인으로 전환한다.

* 성공 기준

  * 정상·오류·수정본 상태가 자동으로 갱신된다.
  * 자동 교정은 오류 수가 줄고 허용 규칙을 만족할 때만 채택된다.
  * 미제출자만 정확히 계산해 리마인드한다.
  * 구조적 실패 3건 통제 평가에서 Agentic Graph 복구율 100%를 재현한다.

### 시나리오 4: Excel 자동 병합 및 기준 파일 일괄 업데이트

* ID: SC-004

* 상황:
  정상 제출된 여러 Excel을 하나로 합치거나, 하나의 기준 Excel 값을 같은 키와 컬럼을 가진 여러 파일에 일괄 반영해야 한다.

* 목표:
  원본을 보존하면서 정상 데이터 병합, 오류 보고서 생성, 기준값 동기화를 자동 수행한다.

* 사전 조건:

  * 대상 파일이 `.xlsx` 형식이며 읽을 수 있어야 한다.
  * 병합 시 검증 계약이, 동기화 시 기준 키와 업데이트할 공통 컬럼이 지정되어야 한다.

* 상세 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 담당 기술/Agent |
| --- | --- | --- | --- |
| 1 | 제출 파일 또는 기준·대상 파일을 선택한다. | 파일·시트·컬럼 구조를 읽는다. | pandas/openpyxl |
| 2 | 병합 또는 기준값 동기화를 선택한다. | 검증 계약 또는 키·공통 컬럼을 확인한다. | Validation/Merge Tool |
| 3 | 실행한다. | 정상 행만 병합하거나 기준 키가 일치하는 행의 공통 값을 대상 파일에 반영한다. | Merge/Sync Tool |
| 4 | 결과를 확인한다. | 처리 건수, 미일치 키, 오류 상세와 결과 파일을 제공한다. | FastAPI, React UI |

* 입력 예시

```text
기준 파일: 기준_조직정보.xlsx
대상 파일: 생산팀.xlsx, 품질팀.xlsx, 물류팀.xlsx
키 컬럼: 사번
업데이트 컬럼: 부서명, 직급, 이메일
```

* 기대 출력

```text
대상 파일: 3개
일치 및 업데이트: 42행
미일치 키: 2건
원본 파일: 변경하지 않음
생성 파일: synced_생산팀.xlsx, synced_품질팀.xlsx, synced_물류팀.xlsx
```

* 성공 기준

  * 정상 데이터만 병합되고 오류 행은 보고서에 분리된다.
  * 기준 키가 일치한 행만 지정 컬럼이 업데이트된다.
  * 원본 파일은 변경되지 않고 결과물이 별도 저장된다.
  * 8파일·130행·정답 오류 26건 평가에서 규칙 검증 F1 1.0을 재현한다.

### 시나리오 5: 최종 통합 검증 및 최초 요청자 회신

* ID: SC-005

* 상황:
  팀장님이 취합 담당자에게 자료를 요청하고, 취합 담당자는 W/G 리더들에게 작성 요청을 발송했다. 모든 작성자의 파일이 정상 제출된 뒤 팀장님에게 최종 취합본을 회신해야 한다. 팀장님이 직접 취합하는 경우에는 최초 요청자와 취합 담당자가 동일하다.

* 목표:
  예상 작성자 전원의 제출을 확인하고 전체 파일을 다시 검증한 뒤, 오류가 0건일 때만 최종 병합본을 최초 요청자·참조자에게 원래 Gmail 대화로 회신한다.

* 상세 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 담당 기술/Agent |
| --- | --- | --- | --- |
| 1 | 별도 행동 없음 | 예상 작성자 이메일 집합과 정상 제출자 집합을 비교한다. | Submission Store, Final Validation Agent |
| 2 | 별도 행동 없음 | 모든 정상 파일을 함께 검사해 파일 간 중복까지 확인한다. | pandas/openpyxl |
| 3 | 오류 건만 확인한다. | 오류가 있으면 완료·병합·팀장님 회신을 차단하고 해당 작성자에게 재작성 요청을 준비한다. | Final Validation/Communication Agent |
| 4 | 필요 시 최종 회신을 검토한다. | 오류가 0건이면 최종 병합본과 완료 회신을 생성한다. | Merge Agent, Completion/Report Agent |
| 5 | 승인 대상이면 제목·본문·수신자를 수정해 발송한다. | 최초 요청자는 To, 원본 참조자는 Cc로 유지하고 병합본을 첨부해 원래 thread에 회신한다. | React UI, Gmail API, Policy Gate |

* 기대 출력

```text
작업번호: SC-20260720-001
작성 대상: 5명
정상 제출: 5명
최종 교차 검증 오류: 0건
최종 병합 행: 128행
회신 대상: 최초 요청자(To) + 원본 참조자(Cc)
첨부: SC-20260720-001_merged.xlsx
상태: 자동 회신 완료 또는 최종 회신 승인 대기
```

* 성공 기준

  * 예상하지 않은 제출자는 완료 인원에 포함하지 않는다.
  * 파일별 검증을 통과했더라도 파일 간 중복이 발견되면 완료하지 않는다.
  * 최종 검증 오류가 0건일 때만 병합본과 완료 회신을 생성한다.
  * 최종 회신은 최초 요청 메일의 thread, 발신자, 참조자를 보존한다.
  * 발송 message-id와 발송 시각이 Collection Job에 저장된다.

### 전체 시나리오 요약

Smart Collect Assistant는 `팀장님 요청 → 메일 확인·분류 → W/G 리더 작성 요청 → 질문 응답 → 제출 검증 → 오류 반려·재제출 → 미제출 리마인드 → 전체 교차 검증 → 최종 병합 → 팀장님 결과 회신`을 하나의 취합 Job으로 연결한다.

LLM은 비정형 메일의 의미, 양식 스키마, 자연어 안내, 상황별 다음 행동을 담당한다. pandas/openpyxl과 정책 코드는 검증, 수신자 제한, 자동발송 허용, 교정 결과 채택을 담당한다. 이 역할 분리로 자동화 범위를 넓히면서도 추측과 오발송 위험을 통제한다.
