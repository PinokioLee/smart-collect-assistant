### Agent 페르소나 및 시스템 프롬프트 Identity

#### 시스템 정의

| 항목 | 정의 내용 |
| --- | --- |
| 시스템명 | Smart Collect Assistant |
| 역할 | Gmail에 도착한 취합 업무를 인지하고, 상황에 맞는 전문 Agent와 Tool을 선택해 후속 업무를 수행하는 자율형 멀티 에이전트 |
| 핵심 원칙 | LLM은 의미와 다음 행동을 판단하고, 결정적 코드는 Excel 검증·수신자·자동발송·교정 채택을 검증 |
| 자동화 목표 | 사람이 모든 건을 처리하지 않고 불명확·고위험·정책 변경 건만 확인 |
| 안전 원칙 | 근거 없는 추측 금지, 원본 파일 보존, 외부 행동 전 Policy Gate, 실패 시 재계획 또는 사람 확인 |
| 결과 원칙 | 분류·판단·도구 실행·오류·재계획·발송 결과를 추적 가능하게 저장 |

#### Agent 구성

| Agent | 책임 | 주요 입력 | 주요 출력/행동 |
| --- | --- | --- | --- |
| Inbox Intake Agent | 새 Gmail 수집, 중복 제거, 메일 계층형 분류 | 제목, 본문, From/Cc, thread, 첨부 | 일반·취합·스팸, 요청·제출·질문·수정·연장, 신뢰도·위험 플래그 |
| Supervisor Agent | 현재 상태와 observation을 보고 다음 행동 선택 | 분류, 신뢰도, 이전 실패, 허용 capability | Worker route 또는 Human Review |
| Security Agent | 스팸·피싱·프롬프트 인젝션 위험 처리 | 위험 플래그, 메일 내용 | 삭제하지 않고 격리 |
| Requirement Analysis Agent | 취합 요청의 구조화 | 제목, 본문 | 목적, 항목, 마감, 대상, 주의사항, 누락 정보 |
| Template Design Agent | 첨부 양식 재사용 또는 신규 양식 설계 | 요구사항, 첨부 Excel | TemplateSpec, Excel 양식, 검증 계약 |
| Communication Agent | 요청·반려·리마인드 메일 작성 | 요구사항, 오류, 수신자, 기존 스타일 | 제목, 본문, 첨부, 발송 요청 |
| Q&A RAG Agent | 작성자 질문에 취합 Job 사실로 답변 | 질문, Gmail thread, Job 정보 | 근거 키가 포함된 답변 또는 승인 요청 |
| Validation Contract Agent | 제출 파일에 적용할 검증 규칙 결정 | Job의 TemplateSpec, 실제 컬럼 | 필수값·날짜·숫자·코드값·중복 규칙 |
| Validation Agent | 제출 Excel 검증 | 파일, 검증 계약 | 정상 행, 오류 행, 행·컬럼별 오류 |
| Self-Correction Agent | 안전한 오류 교정 제안과 재검증 | 날짜·코드값 오류 | 교정 후보, 재검증 결과, 채택/기각 |
| Merge/Sync Agent | 정상 파일 병합 및 기준값 동기화 | 정상 제출, 기준 파일, 키·공통 컬럼 | 최종 병합본 또는 동기화 파일 |
| Deadline Agent | 마감·미제출자 관리 | Job, 수신자, 제출 현황, 현재 시각 | 리마인드 또는 연장 승인 요청 |
| Final Validation Agent | 작성자 집합과 전체 파일을 최종 교차 검증 | 예상 작성자, 정상 제출, 전체 Excel | 완료 허용 또는 재작성 요청 |
| Completion/Report Agent | 최초 요청자에게 최종 결과 회신 | 최종 검증, 병합본, 원본 요청 thread | 완료 메일 초안·자동발송·승인 대기 |
| Human Review | 자동화 경계를 넘는 건 처리 | 낮은 신뢰도, 정책 변경, 구조 실패 | 승인·수정·반려 결정 |

#### 핵심 System Prompt

```text
당신은 Smart Collect Inbox Supervisor Agent다.

목표
- 현재 메일과 취합 Job 상태를 이해하고 허용된 행동 중 다음 행동 하나를 선택한다.
- 이전 Worker가 실패했다면 observation을 읽고 안전한 대안을 선택한다.

원칙
1. 일반 메일은 보관하고 스팸·위험 메일은 삭제하지 말고 격리한다.
2. 신규 요청은 요구사항·양식·수신자·위험도를 확인한 뒤 요청 Worker로 보낸다.
3. 제출·수정본은 검증 Worker, 질문은 Q&A Worker, 연장 요청은 승인 Worker로 보낸다.
4. Job, 첨부, 수신자, 근거가 불명확하면 추측하지 말고 Human Review를 선택한다.
5. 일시 오류만 한 번 재시도하고 구조적 실패나 재실패는 Human Review로 전환한다.
6. 외부 발송은 별도 Policy Gate가 허용한 경우에만 실행한다.
7. 지정된 JSON 스키마로 action과 reason만 반환한다.
```


### 최종 핵심 시나리오

메일 도착부터 취합 완료까지 하나의 Job으로 연결한다.

```text
Gmail 새 메일 / APScheduler / 지금 실행
  → 메일 3분류 + 취합 5개 의도 분류
  → Supervisor가 다음 Worker 선택
  → 요청: 요구 분석 → 양식 결정 → 안내 메일 → 자동발송/승인
  → 질문: thread/Job 연결 → 사실 검색 → 자동답변/승인
  → 제출·수정: 규칙 검증 → 제한적 교정 → 반려 또는 정상 등록
  → 마감: 미제출자 계산 → 리마인드
  → 전원 정상: 전체 파일 교차 검증 → 오류 0건만 최종 Excel 병합
  → Completion/Report Agent: 최초 요청자·참조자에게 병합본 회신
  → 실패: observation → 일시 오류 1회 재시도 → 사람 확인
```

#### 입력 데이터

| 입력 | 내용 |
| --- | --- |
| Gmail message | id, threadId, 제목, 본문, 발신자, 수신자, 참조자, 수신 시각 |
| 첨부 | `.xlsx`, `.xls`, `.csv` 파일 경로와 메타데이터 |
| 조직도 | 이름, 부서, 이메일 CSV 또는 JSON |
| 스케줄 | 활성화, 실행 시각 목록, 타임존 |
| 운영 정책 | 자동발송 활성화, 최소 신뢰도, 허용 도메인, mock/gmail 모드 |
| 취합 Job | Job ID, 원본 thread, 최초 요청자·참조자, 작성자, 요구사항, 양식, 검증 계약, 마감 |

#### 출력 데이터

| 출력 | 내용 |
| --- | --- |
| Inbox record | 분류, 의도, 신뢰도, 위험, 상태, 처리 결과 |
| Collection Job | 요청부터 제출·완료까지 연결한 업무 상태 |
| Draft/Send result | 요청·반려·질문 답변·리마인드 메일과 발송 ID |
| Excel artifacts | 신규 양식, 교정본, 병합본, 오류 보고서, 동기화 결과 |
| Agent Action Log | Agent, action, outcome, reason, observation, timestamp |
| Langfuse trace | 메일 이벤트 단위 Trace와 LLM 호출별 Generation |
| Final reply | 전체 검증 결과, 병합본, 최초 요청자 회신 상태·message-id |

### 워크플로우 및 오케스트레이션 Workflow & Logic

#### Step 1. Event Intake & Classification

1. APScheduler 또는 `지금 실행`이 Gmail Adapter를 호출한다.
2. 최근 새 메일을 읽고 message-id로 중복 처리를 막는다.
3. LLM이 상위 category와 세부 intent, confidence, risk_flags, reason을 JSON으로 반환한다.
4. Gmail thread가 활성 취합 Job과 정확히 연결된 짧은 회신이면 thread 문맥으로 제출·질문·수정·연장 의도를 보강한다. 스팸 판정은 덮어쓰지 않는다.
5. LLM 장애 시 휴리스틱 폴백을 사용하고 출처를 기록한다.

#### Step 2. Supervisor Planning & Policy Route

Supervisor에게는 한 개의 고정 경로가 아니라 현재 분류에서 가능한 Worker capability 목록을 제공한다. LLM이 다음 행동을 선택한 뒤 Route/Intent Policy Gate가 의도와 맞지 않는 Worker 실행을 차단한다.

| 분류/상태 | 허용 행동 | 기본 행동 |
| --- | --- | --- |
| general | General Mail Agent | 별도 행동 없이 보관 |
| spam | Security Agent | 격리 |
| collection/request | Request Worker, Human Review | 요구분석·양식·메일 준비 |
| collection/submission | Submission Worker, Human Review | 검증·반려 또는 병합 |
| collection/correction | Submission Worker, Human Review | 수정본 재검증 |
| collection/question | Q&A Worker, Human Review | 근거 기반 답변 |
| collection/extension | Extension Worker, Human Review | 사람 승인 |

#### Step 3. Worker Execution

* Request Worker
  1. 요구사항 구조화
  2. 첨부 양식 재사용 또는 TemplateSpec 생성
  3. TemplateSpec에서 Excel 양식과 검증 계약을 함께 생성
  4. 명시 대상자 또는 원본 From+Cc로 수신자 구성
  5. 과거 발송 스타일과 요구사항을 반영해 메일 작성
  6. Policy Gate 결과에 따라 발송 또는 승인 대기

* Submission/Correction Worker
  1. thread/Job ID로 Collection Job 연결
  2. 검증 계약 선택
  3. 결정적 규칙으로 Excel 검증
  4. LLM Self-Correction 제안 → 허용값·날짜 게이트 → 재검증
  5. 오류 감소 시 교정본 채택, 아니면 원본 유지
  6. 오류가 남으면 사실 기반 반려, 정상이면 제출 등록 및 병합

* Q&A Worker
  1. 질문자와 Job 연결 검증
  2. 마감·작성 항목·검증 규칙·가이드에서 근거 검색
  3. 근거 키와 함께 답변 생성
  4. 정책 변경 여부 확인 후 자동답변 또는 승인 대기

* Deadline Worker
  1. 수신자와 정상 제출자 집합 비교
  2. 마감 임박 미제출자 계산
  3. 리마인드 작성
  4. 발송 정책 적용

* Final Validation/Completion Worker
  1. 예상 작성자 이메일 집합과 정상 제출자 집합 비교
  2. 전체 제출 파일을 묶어 파일 간 중복까지 재검증
  3. 오류가 있으면 완료를 차단하고 해당 작성자에게 재작성 요청
  4. 오류가 0건이면 최종 병합본 생성
  5. LLM이 확정 사실만 사용해 완료 회신 작성
  6. 최초 요청자·참조자에게 원래 Gmail thread로 자동발송 또는 승인 대기

#### Step 4. Observation & Replanning

Worker는 성공/실패와 observation을 반환한다. 성공은 종료하고 실패는 Supervisor로 되돌린다.

| 실패 유형 | 처리 |
| --- | --- |
| timeout, 429, 503, connection reset 등 일시 오류 | 동일 업무 1회 재시도 가능 |
| Job 미확인 | 임의 연결하지 않고 Human Review |
| 첨부 유실·손상 파일 | 재실행으로 해결되지 않으므로 Human Review |
| 재시도 후 재실패 | Human Review |
| Worker/intent 불일치 | Policy Gate가 차단하고 Human Review |

#### Step 5. External Action Gate

자동발송은 LLM의 “단순 업무” 판단 하나로 결정하지 않는다. 아래 조건을 모두 통과해야 한다.

* 지원되는 행동: 취합 요청, 검증 반려, 단순 질문, 마감 리마인드
* 분류 신뢰도 `AUTO_SEND_MIN_CONFIDENCE` 이상
* 마감·필수 항목·첨부가 명확하고 누락 정보 없음
* 수신자 이메일이 유효하고 허용 도메인에 포함
* 실제 첨부 가능한 양식 파일 존재
* 질문자는 해당 Job의 대상자이며 thread 또는 Job ID가 일치
* RAG 근거 부족, 프롬프트 인젝션, 정책 변경 위험 없음
* Supervisor가 사람 확인 불필요로 판단
* `AUTO_SEND_ENABLED=true`

하나라도 충족하지 못하면 발송하지 않고 승인 큐로 이동한다.


### 상태 관리

#### InboxAgentState

| 상태명 | 설명 |
| --- | --- |
| event_id | Gmail message-id 기반 이벤트 식별자 |
| message | 제목, 본문, 주소, thread, 첨부를 포함한 메일 객체 |
| classification | category, intent, confidence, tier, risks, source |
| route | Supervisor가 선택한 다음 Worker |
| attempt | Worker 실행 횟수 |
| outcome | classified, success, failure |
| observation | Worker 실패 사유와 오류 정보 |
| job_id | 연결된 Collection Job ID |
| terminal | 종료 여부 |

#### Collection Job 및 Excel 처리 State

| 상태명 | 설명 |
| --- | --- |
| request_id/job_id | 취합 요청 한 건의 고유 ID |

### LangGraph Node / Edge 흐름

```text
START
  ↓
intake
  ↓
supervisor ─────────────────────────────────────────┐
  ├─ general → END                                 │
  ├─ quarantine → END                              │
  ├─ request_worker ─┐                             │
  ├─ submission_worker ─┤                          │
  ├─ qa_worker ────────┤→ observe ─ success → END │
  ├─ extension_worker ─┘           └ failure ──────┘
  └─ human_review → END
```

고정 Workflow와의 차이는 Worker 실패를 단순 오류로 종료하지 않고 observation을 상태에 넣어 Supervisor가 다시 판단한다는 점이다. 동일 Worker capability 14개 시나리오에서 Fixed 78.57%, Agentic 100%의 성공률을 기록했다.

### 지식 베이스 및 메모리 전략 Context & Memory

#### RAG 적용 전략

RAG는 모든 단계에 사용하지 않는다. 외부 또는 과거 사실을 참고해야 하는 자연어 생성에만 적용하고, Excel 규칙 검증에는 사용하지 않는다.

| 기능 | RAG | 이유 |
| --- | --- | --- |
| 메일 분류 | 미적용 | 현재 메일 문맥만으로 판단하고 평가 가능해야 함 |
| 요청 메일 작성 | 스타일 RAG | 담당자의 과거 인사말·문장 길이·마무리 표현 반영 |
| 작성자 질문 답변 | Job grounding/RAG | 원래 마감·필드·허용값에 근거해 환각 방지 |
| Excel 검증 | 미적용 | 저장된 검증 계약과 실제 셀 값으로 재현 가능하게 처리 |
| 반려 메일 | 오류 사실 grounding | 검증 결과에 없는 오류를 만들지 않도록 제한 |

#### 메모리 및 연결 키

* Gmail `threadId`: 같은 대화의 질문·제출·수정본 연결
* `[SC-...]` Job ID: 제목이 유지되는 회신과 업무 상태 연결
* SQLite Collection Job: 요구사항, 양식, 수신자, 마감, 검증 계약 저장
* Submission: 제출자, 파일, 상태, 오류 저장
* Agent Action Log: 판단, 도구, observation, 재계획 저장

thread와 Job ID로 하나의 취합 건을 확정할 수 없으면 가장 비슷한 Job을 추측해 연결하지 않는다.

### 최종 요약

Smart Collect Assistant는 고정 순서의 자동화가 아니라 메일 이벤트와 현재 업무 상태를 보고 전문 Worker를 선택하는 멀티 에이전트다. LLM은 분류, 요구사항·양식 설계, 라우팅, 안내·반려·질문 답변에 사용하고, 결정적 Tool은 Excel 검증, 수신자·발송 정책, 교정 채택, 병합을 담당한다.

핵심 고도화는 Worker 실패를 observation으로 되돌려 Supervisor가 재계획하는 구조, Gmail thread와 Job ID를 이용한 업무 문맥 유지, 자동발송과 사람 승인을 가르는 Policy Gate, LLM 호출을 Langfuse Generation으로 추적하는 관찰 가능성이다. 이를 통해 자동화 범위와 운영 안전성을 함께 확보했다.
