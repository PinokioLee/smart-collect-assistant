테스트 및 고도화
PoC 단계에서 구현된 핵심 모듈의 성능을 평가하고, 한계점을 극복하기 위해 적용한 기술적 최적화 및 테스트 내역을 상세히 기술합니다.

## **주요 문제 해결 및 기술 리서치 (테스트 단계)**

테스트 과정에서 발견된 엣지 케이스(Edge Case)와 성능 병목을 해결하기 위해 리서치하고 적용한 방법을 기록합니다.

| **이슈 구분** | **문제 상황 및 원인** | **리서치 및 해결 과정 (Reference & Solution)** |
| --- | --- | --- |
| **품질/환각** | 단순 LLM에게 엑셀 검증까지 맡기면 실제 데이터에 없는 오류 사유를 그럴듯하게 생성하거나, 동일 입력에서도 설명이 달라질 위험이 있었다. 엑셀 검증은 업무 결과 파일에 직접 반영되므로 자연어 판단보다 재현성이 중요했다. | **리서치:** LLM 기반 판단과 규칙 기반 데이터 검증의 역할을 분리하는 방식을 검토하였다. **적용:** 메일 분석은 Azure OpenAI 또는 휴리스틱으로 처리하되, 엑셀 검증/병합은 pandas/openpyxl 기반 결정론적 규칙으로 고정하였다. 그 결과 라벨링 데이터셋 10개 오류 기준 Precision/Recall/F1 모두 100%, 10회 반복 오류 수 표준편차 0.0000을 달성하였다. |
| **스키마 드리프트/오탐** | 메일에서 추출된 작성 항목을 모두 필수 컬럼으로 강제하는 Strict 규칙만 사용하면, 실제 양식에서 일부 설명성 컬럼이 빠진 경우 전체 파일에 "필수 컬럼 누락" 오탐이 발생할 수 있었다. | **리서치:** Tree of Thoughts 방식처럼 여러 후보 규칙을 생성하고, 실제 업로드 파일 컬럼과의 coverage/penalty를 비교하는 방식을 검토하였다. **적용:** A.Strict, B.Balanced, C.Loose 3개 후보를 생성하고 실제 컬럼과 대조하여 선택하도록 구현하였다. 테스트에서 `요청사유` 컬럼이 빠진 드리프트 상황에서 B.Balanced가 선택되어 과잉 필수규칙 오탐 1건을 회피하였다. |
| **자가 교정 범위** | 모든 오류를 자동 수정하면 데이터 생성/추측이 발생할 수 있다. 특히 필수값 누락과 중복 데이터는 작성자의 의도를 알 수 없어 자동 보정하면 위험했다. | **리서치:** Self-Refine/Self-Correction 패턴을 검토하되, 업무 데이터에서는 "안전하게 정규화 가능한 오류"만 교정 대상으로 제한하였다. **적용:** 날짜 형식과 코드값 매핑만 자동 교정하고, 필수값 누락/중복은 재제출 대상으로 남겼다. 벤치마크에서 교정 가능 오류 6건 중 6건을 교정하여 자동 교정율 100%, 잔여 재작업 60% 감소를 확인하였다. |
| **속도/지연** | 수작업 방식은 파일을 열고, 행별로 누락/날짜/코드/중복을 확인하고, 오류를 별도 보고서에 기록해야 한다. 파일 수와 행 수가 늘어날수록 선형으로 시간이 증가한다. | **리서치:** pandas DataFrame 기반 일괄 검증과 openpyxl 기반 결과 파일 생성을 적용하였다. **적용:** 4개 파일/46행 라벨링 데이터셋에서 로드+검증+병합 자동 처리 약 0.08초, 수동 추정 15.5분 대비 99.99% 단축을 측정하였다. |
| **보안/가드레일** | 취합 메일과 엑셀에는 이름, 이메일, 업무 요청 내용 등 민감 정보가 포함될 수 있다. 또한 메일 발송은 실제 수신자에게 영향을 주므로 자동 발송은 오발송 리스크가 있다. | **리서치:** Human-in-the-loop 승인 구조와 원본 보존 정책을 검토하였다. **적용:** PoC 기본값은 mock 발송이며, 실제 Gmail 발송은 사용자가 UI에서 승인하고 OAuth credentials를 설정한 경우에만 가능하다. 엑셀 원본은 수정하지 않고 병합 파일, 오류 보고서, 업데이트 파일을 별도 경로에 생성한다. 테스트에서 공통항목 일괄 수정 후 원본 파일에 신규 컬럼이 생기지 않는 것을 검증하였다. |
| **시연 안정성** | Windows PowerShell 기본 CP949 콘솔에서 블록 문자, 이모지, 일부 특수 기호 출력 시 `UnicodeEncodeError`가 발생하여 demo/benchmark 명령이 중단될 수 있었다. | **리서치:** 시연 환경에서는 기능보다 콘솔 호환성이 중요하므로 출력 문자를 ASCII 중심으로 단순화하였다. **적용:** `backend/cli.py`, `backend/smart_collect/benchmark.py`의 장식 문자와 화살표를 ASCII로 변경하여 별도 인코딩 설정 없이도 실행 가능하도록 고도화하였다. |

## 1. LLM 답변 품질 평가 및 개선

| 항목 | 내용 |
| --- | --- |
| 평가 대상 기능 | 취합 요청 메일 분석 결과의 구조화, 엑셀 검증 규칙 생성, 최종 오류 검증 결과의 신뢰성 |
| 평가 방식 | LLM이 직접 엑셀 오류를 판단하지 않도록 역할을 분리하고, 라벨링된 벤치마크 데이터셋 4개 파일/46행/정답 오류 10건을 구축하여 검출 결과를 비교하였다. |
| 초기 문제 | 단순 LLM 기반 검증은 설명 생성에는 유리하지만, 행 번호/컬럼/오류 유형이 항상 동일하게 재현된다는 보장이 부족했다. 발표 및 실무 적용에서는 "그럴듯한 설명"보다 "같은 입력이면 같은 오류 보고서"가 더 중요했다. |
| 개선 조치 | 메일 분석은 LLM 또는 휴리스틱으로 처리하고, 엑셀 검증은 `validate_required_fields`, `validate_date_format`, `validate_code_values`, `validate_duplicates` 네 가지 결정론적 함수로 분리하였다. 출력은 Pydantic 모델과 JSON 구조로 고정하였다. |
| 평가 결과 | 정답 오류 10건 중 10건 검출, 오탐 0건, 미탐 0건으로 Precision 100%, Recall 100%, F1 100%를 달성하였다. |
| 개선 후 의미 | LLM 환각 가능성이 높은 영역을 검증 로직에서 제거하고, LLM은 메일 분석/가이드 초안처럼 사람 검토가 가능한 영역에 제한적으로 사용하도록 역할을 재정의하였다. |

## 4. 기타 문제 해결 사례

### 4.1 Tree of Thoughts 기반 검증 규칙 선택

단순히 메일에서 추출된 모든 항목을 필수 컬럼으로 강제하면 실제 부서별 양식이 조금만 달라도 오탐이 발생한다. 이를 해결하기 위해 검증 규칙 후보를 세 가지로 생성하였다.

| 후보 | 특징 | 장점 | 위험 |
| --- | --- | --- | --- |
| A.Strict | 추출된 작성 항목을 최대한 필수로 요구 | 누락 방지에 강함 | 양식 변형 시 오탐 가능 |
| B.Balanced | 주의사항과 실제 컬럼을 함께 고려 | 업무 기준과 현실 양식의 균형 | 일부 설명성 항목을 필수에서 제외할 수 있음 |
| C.Loose | 최소 필수 항목 중심 | 오탐 감소 | 누락 검출 약화 가능 |

벤치마크의 스키마 드리프트 상황에서 Strict 후보는 실제 파일에 없는 `요청사유`를 필수로 요구해 오탐을 만들 수 있었지만, ToT 선택 결과 B.Balanced가 선택되어 오탐 1건을 회피하였다.

### 4.2 Self-Correction 적용 기준

Self-Correction은 모든 오류를 자동으로 고치는 기능이 아니라, "업무 의미를 추측하지 않고 안전하게 정규화할 수 있는 오류"만 줄이는 기능으로 설계하였다.

| 오류 유형 | 자동 교정 여부 | 이유 |
| --- | --- | --- |
| 날짜 형식 오류 | 가능 | `2026/06/05`, `2026.06.05`처럼 의미가 명확한 값을 `2026-06-05`로 정규화 가능 |
| 코드값 오류 | 가능 | `매우 급함`, `긴급` 등은 사전 정의된 매핑으로 `상`에 대응 가능 |
| 필수값 누락 | 불가 | 작성자 의도를 추측해야 하므로 자동 생성 금지 |
| 중복 데이터 | 불가 | 어느 행을 유지할지 업무 판단이 필요하므로 재제출/사용자 확인 필요 |

이 기준으로 벤치마크에서 오류 10건 중 교정 가능 6건을 모두 교정하였고, 잔여 오류는 보고서에 남겨 사용자 판단 대상으로 분리하였다.
