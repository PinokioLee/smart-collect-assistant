정량 평가, 예외 처리, 안전성 및 Agentic 구조 개선


## 주요 문제 해결 및 기술 리서치 (테스트 단계)

테스트는 단순 성공 화면 확인이 아니라 다음 세 질문에 답하도록 설계했다.

1. 키워드 규칙보다 LLM이 다양한 메일 표현을 더 정확히 이해하는가?
2. 동일 Worker를 사용할 때 Observation Loop가 있는 Agentic Graph가 고정 흐름보다 예외를 더 안전하게 처리하는가?
3. Excel 검증에도 LLM을 써야 하는가, 아니면 결정적 규칙이 더 적합한가?

| 이슈 | 테스트에서 확인한 문제 | 고도화 내용 |
| --- | --- | --- |
| 휴리스틱 분류 한계 | “자료 부탁드립니다” 같은 우회 표현과 수정·연장 의도를 놓침 | 24개 난이도 평가셋과 Azure LLM Structured Output 도입 |
| 형식적인 Multi-Agent | 고정 순서만 실행하면 LangGraph가 단순 함수 체인과 차이가 없음 | Supervisor 선택, Policy Route, Worker Observation, 재계획 구현 |
| 구조 실패 처리 | Job 없음·첨부 유실·손상 파일에서 Fixed Workflow가 오류 종료 | 재시도 가능성 판정 후 Human Review로 안전 복구 |
| Excel 검증 LLM 남용 | 동일 오류 검출에 수 초가 걸리고 비용·재현성 부담 | pandas/openpyxl 규칙 엔진으로 분리 |
| 자동 교정 위험 | LLM이 빈 값이나 자유서술을 추측할 수 있음 | 날짜·코드값만 허용, 코드 게이트와 전체 재검증 적용 |
| 질문 답변 환각 | 원래 취합 기준에 없는 내용을 답할 수 있음 | Gmail thread/Job 연결, 근거 키 기록, 정책 변경 질문 승인 처리 |
| 자동발송 위험 | LLM confidence만으로 오발송을 막기 어려움 | 수신자·도메인·첨부·근거·위험·운영 설정 Policy Gate |
| LLM 사용 추적 부족 | 최종 결과만으로 어떤 호출이 영향을 줬는지 확인 어려움 | 이벤트 Trace와 호출별 Langfuse Generation 추가 |


## 1. LLM 답변 품질 평가 및 개선

### 1.1 메일 의미 분류 비교

* 평가 데이터: 고정 난이도 평가셋 24건
* 구성: 일반, 스팸, 요청, 제출, 질문, 수정, 연장
* 특징: 단순 키워드가 없는 우회 표현과 문맥 의존 표현 포함
* 판정 기준: 상위 category와 세부 intent가 모두 일치해야 정답

| 방식 | 상위 분류 정확도 | category+intent exact match | 중앙 지연 | 실제 LLM 응답 |
| --- | ---: | ---: | ---: | ---: |
| 휴리스틱 | 54.17% | 45.83% | 0.017ms | 0 |
| Azure LLM | 100% | 100% | 2,457.98ms | 24/24 |

#### 해석

* 휴리스틱은 매우 빠르지만 표현이 달라지면 일반 메일로 오분류하거나 수정·연장 의도를 구분하지 못했다.
* LLM은 평가셋 전체를 맞췄지만 중앙 지연이 약 2.46초였다.
* 따라서 새 메일의 의미 판단에는 LLM을 사용하고, 장애 시 휴리스틱으로 폴백하되 출처를 기록한다.
* 100%는 내부 고정 평가셋 결과다. 운영 전에는 비식별 실제 메일의 블라인드 평가가 필요하다.

### 1.2 구조화 출력과 방어적 파싱

LLM 결과는 자유 텍스트가 아니라 JSON Schema로 제한했다.

* category와 intent 허용값 제한
* confidence 0~1 범위 제한
* required와 additionalProperties 설정
* 잘못된 값은 자동 행동에 사용하지 않고 검토 대상으로 전환
* LLM 실패 시 휴리스틱 폴백과 `source` 기록

이 방식으로 자연어 파싱 실패가 업무 실행 오류로 바로 이어지는 것을 줄였다.

### 1.3 생성 메일 Grounding

요청·반려·질문 답변은 생성 후 다음을 다시 확인한다.

| 검사 | 근거 |
| --- | --- |
| 마감 | 원본 요구사항 또는 Collection Job |
| 작성 항목·허용값 | TemplateSpec/ValidationRule |
| 수신자 | 원본 From+Cc 또는 조직도 결과 |
| 첨부 | 실제 존재하는 양식 파일 |
| 오류 반려 | 코드 검증의 행·컬럼별 오류 |
| 질문 답변 | Job에 저장된 사실과 사용 근거 키 |

근거가 부족하거나 정책을 바꾸는 내용이면 자동발송하지 않는다.


## 2. 성능 및 비용 최적화

### 2.1 Excel 검증 기술 비교

* 데이터: 8개 Excel, 130행, 정답 오류 26건
* Direct LLM: 3회 실행
* 규칙 엔진: 10회 실행

| 방식 | Precision | Recall | F1 | 평균 시간 | 표준편차 |
| --- | ---: | ---: | ---: | ---: | ---: |
| Direct LLM | 1.0 | 1.0 | 1.0 | 3,933.7ms | 84.3ms |
| pandas/openpyxl 규칙 | 1.0 | 1.0 | 1.0 | 21.12ms | 4.145ms |

두 방식의 정확도가 같았고 규칙 엔진이 약 186배 빨랐다. 따라서 셀 단위 검증에는 LLM을 사용하지 않는다. 이 선택은 “AI 과제이므로 모든 단계에 LLM을 넣는 방식”보다 비용, 속도, 재현성 측면에서 타당하다.

### 2.2 선택적 LLM 호출

| 단계 | LLM 사용 | 이유 |
| --- | --- | --- |
| 메일 분류 | 사용 | 표현과 문맥이 다양함 |
| Supervisor 라우팅 | 사용 | 현재 상태와 실패 observation을 함께 판단 |
| 자연어 요구사항→양식 | 사용 | 고정 규칙만으로 다양한 업무 필드를 설계하기 어려움 |
| 안내·반려·질문 답변 | 사용 | 사람이 이해할 자연어와 문맥 반영 필요 |
| Excel 셀 검증 | 미사용 | 결정적 규칙이 동일 정확도에서 훨씬 빠름 |
| 수신자·도메인·첨부 Gate | 미사용 | 외부 행동은 재현 가능한 정책으로 통제해야 함 |
| 교정 후보 채택 | 미사용 | 오류 감소와 허용값을 코드로 확인해야 함 |

### 2.3 관찰 가능성을 이용한 비용 관리

Langfuse Generation에 모델, 입력, 출력, token usage, latency, error를 기록한다. 이를 통해 어떤 Agent의 프롬프트가 비용과 지연을 유발하는지 확인할 수 있다. Langfuse 장애가 본 업무를 막지 않도록 관측 경로는 선택적으로 동작한다.

## 4. 기타 문제 해결 사례

### 4.1 검증 규칙 후보 선택

배포한 TemplateSpec이 있는 경우에는 그 계약을 그대로 사용한다. 외부 또는 레거시 파일처럼 확정 계약이 없을 때만 여러 검증 규칙 후보를 비교한다.

| 후보 | 특징 | 위험/보완 |
| --- | --- | --- |
| Strict | 추출한 모든 항목을 필수로 사용 | 실제 파일에 없는 컬럼 강제로 오탐 가능 |
| Balanced | 실제 컬럼과 요구사항의 교집합 중심 | 일부 선택 규칙이 약해질 수 있음 |
| Minimal | 최소 공통 컬럼만 검사 | 오류 누락 가능 |

각 후보의 실제 컬럼 coverage를 높이고 존재하지 않는 필수 컬럼 penalty를 낮추는 점수로 선택한다. 후보와 점수, 최종 선택을 추론 로그에 남긴다.

### 4.2 Self-Correction 적용 기준

| 오류 유형 | 자동 교정 | 기준 |
| --- | --- | --- |
| 날짜 표기 변형 | 가능 | ISO 날짜로 변환 가능하고 재검증 통과 |
| 코드값 표기 변형 | 제한적 가능 | 허용값 중 하나로 확정되고 코드 Gate 통과 |
| 필수값 누락 | 불가 | 업무 사실을 추측해야 함 |
| 숫자 내용 오류 | 불가 | 올바른 값을 시스템이 알 수 없음 |
| 자유서술 내용 | 불가 | 의미 변경 위험 |
| 중복 데이터 | 불가 | 어느 행을 삭제할지 업무 판단 필요 |

교정본은 원본과 별도 저장하며, 전체 재검증에서 오류 수가 감소할 때만 채택한다.

### 4.3 Agentic 효과 비교

비교의 1차 기준은 기능 범위가 같은 `llm_fixed_workflow`와 `agentic_supervisor_graph`다. 초기 `rule_sequential`은 기능 범위가 작아 Agentic 효과의 인과 비교에는 사용하지 않는다.

| 지표 | Fixed LLM | Agentic Supervisor | 차이 |
| --- | ---: | ---: | ---: |
| 시나리오 수 | 14 | 14 | 동일 |
| Workflow 성공률 | 78.57% | 100% | +21.43%p |
| 자율 해결률 | 71.43% | 71.43% | 동일; Agentic은 실패를 안전한 사람 확인으로 복구 |
| 구조 실패 복구율 | 0%, n=3 | 100%, n=3 | +100%p |
| 사람 확인 건수 | 1 | 4 | 구조 실패 3건을 오류 종료 대신 의도적으로 사람에게 전달 |
| E2E 중앙 지연 | 2,669.13ms | 5,837.94ms | 추가 Supervisor 판단으로 약 3.17초 증가 |
| 잘못된 허용 판단 | 0 | 0 | mock·자동발송 OFF 통제 환경 |

Agentic의 가치는 모든 일을 자동으로 끝내는 비율을 무조건 높이는 데 있지 않다. 처리할 수 없는 건을 오류로 버리지 않고, 실패 이유를 관찰해 재시도 또는 사람 확인이라는 올바른 다음 상태로 바꾸는 데 있다. 그 대가로 중앙 처리 지연이 약 3.17초 늘었다. 메일 기반 비동기 업무에서는 이 지연보다 실패 복구와 오발송 방지가 중요하다고 판단했다.

### 4.4 사람 대비 ROI 측정 원칙

사람의 Before 시간은 아직 스톱워치 실측 데이터가 없으므로 시간 절감률을 확정하지 않는다.

* `scripts/manual_roi_timer.py`로 같은 14개 시나리오를 최소 3회 측정
* 가능하면 2명 이상 참여
* 메일 확인, 파일 열기, 검증, 반려/승인 판단의 능동 시간만 측정
* `ESTIMATE` 또는 `추정` 표시는 실측에서 제외
* 유효 CSV가 있을 때만 `roi_claim_available=true`와 ROI 계산

이 원칙은 발표 수치를 과장하지 않고 실제 운영 효과를 추후 재현하기 위한 것이다.
