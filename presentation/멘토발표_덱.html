Smart Collect
Assistant

여러 부서에서 받은 엑셀 취합 파일을 자동으로 검증·병합하고, 취합 요청 메일을 분석해 검증 기준을 세우는 멀티 에이전트 업무 자동화 시스템.

AI에게는 메일을 읽게 하고, 엑셀 검사는 정확한 계산기 같은 코드에게 맡겼다. LangGraph는 에이전트들이 어떤 순서로 일할지 정하는 반장 역할이다.
발표자 이형진 · 08079 React + TypeScript FastAPI LangGraph 멀티에이전트

문제 정의

매달 반복되는 엑셀 취합,
사람이 손으로 검사한다

지금의 방식

부서 여러 곳에서 제각각 양식으로 회신 → 필수값 누락·날짜 형식·중복을 눈으로 확인 → 정상 행만 골라 수기로 병합 → 오류는 다시 회신 요청.

왜 자동화인가

같은 일을 매번, 실수 없이 반복해야 한다. 판단은 거의 없고 규칙은 명확하다. 자동화의 이득이 가장 큰 전형적 반복 업무다.

필수값 누락 날짜 형식 흔들림 파일 간 중복 양식 제각각 취합 1건 ≈ 수기 15.5분(추정)

한 장 요약 · 처리 흐름

지저분한 입력 → 깨끗한 출력

취합 요청 메일 1건→ 작성 항목·기준 추출→ 제출 엑셀 N개 업로드→ 5종 규칙 검증→ 정상만 병합→ 오류 보고서→ 요약

입력은 자연어 메일 + 형식이 어긋난 엑셀들. 출력은 병합본 · 오류 보고서 · 요약. 원본은 손대지 않고 결과는 모두 별도 파일로 만든다.

핵심 설계 철학

두 종류의 일을 분리했다

LLM이 잘하는 일 언어 이해

  • 자연어 취합 메일 이해
  • 작성 항목 · 마감 · 주의사항 추출
  • 작성 가이드 · 요청 메일 초안
  • “틀려도 되고, 창의적이어도 되는” 영역

규칙이 잘하는 일 결정론

  • 엑셀 행 단위 검증 (pandas/openpyxl)
  • 정상 행 병합 · 오류 보고서
  • “항상 같아야 하고, 재현되어야 하는” 영역
  • LLM이 행의 유효성을 판단하지 않는다
LangGraph = 반장. 나뉜 역할의 에이전트들이 어떤 순서로 협업할지 정하고, 상태(AgentState)를 넘겨가며 일을 진행시킨다.

Self-Refine

Self-Correction

LLM이 교정값을 제안하면 코드가 결정론 게이트로 검증. 날짜·코드값만 안전 교정하고 필수값·중복은 손대지 않는다.

핵심 논지

왜 엑셀 검증을 LLM에 맡기지 않았나

LLM으로 검증하면 불안정

  • 행 번호·오류 유형이 실행마다 흔들릴 수 있음
  • 같은 파일에도 다른 답이 나올 위험
  • 비용·지연 발생

규칙으로 검증하면 재현

  • 같은 입력 → 같은 행·같은 오류 유형 보장
  • 결정론 게이트가 채택까지 관장
  • 발표·감사에서 근거를 그대로 재현
검증은 창의적인 답이 아니라 정확하고 재현 가능한 답이 필요하다. 그래서 LLM은 값의 의미를 제안하고, 코드가 검증·확정한다.

멘토 예상 Q&A · 데모

질문에 이렇게 답한다

Q왜 검증을 규칙으로?

A재현성. 같은 입력이면 같은 행·같은 오류가 나와야 하기 때문.

QGmail을 실제로 읽나?

A기본은 mock 수신함, credentials 설정 시 실제 gmail.readonly. 분류·초안까지만 하고 자동 발송은 안 함(사람 승인).

Q담당자 주소는 어떻게?

ARAG로 추측하지 않고 조직도(DB)에서만 가져온다. 잘못된 발송을 막기 위함.

QRAG는 무엇을 하나?

A메타데이터 필터·쿼리 재작성·근거 검증까지 하는 고급 RAG. 임베딩 없이, 근거 없는 항목은 플래그.
