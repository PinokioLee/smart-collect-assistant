“안녕하세요, 08079 이형진입니다. 제가 만든 Smart Collect Assistant를 소개하겠습니다. 한 문장으로 요약하면 — 여러 부서에서 받은 엑셀을 자동으로 검증하고 병합해 주는 멀티 에이전트 시스템입니다. 이 프로젝트의 핵심 아이디어는 화면의 이 문장입니다: ‘AI에게는 메일을 읽게 하고, 엑셀 검사는 정확한 계산기 같은 코드에게 맡겼다.’ 이 역할 분리가 오늘 발표의 중심 주제입니다.”

팁

인용구는 천천히 또박또박 읽어 강조하세요. 첫 슬라이드에서 오늘의 주제어(‘분리’)를 각인시키는 게 목표.

02문제 정의≈ 50초
화면

현재의 수기 방식 · 왜 자동화인가 · 오류 유형 칩(누락·형식·중복·15.5분).

대본

“먼저 어떤 문제를 풀려고 했는지 말씀드리겠습니다. 회사에서는 매달 각 부서에 ‘이런 항목을 작성해서 보내주세요’라는 취합 요청 메일을 보냅니다. 그러면 부서마다 제각각 양식으로 엑셀을 회신하죠. 담당자는 이걸 하나하나 열어 필수값이 비었는지, 날짜 형식이 맞는지, 중복은 없는지 눈으로 확인하고, 정상인 것만 골라 손으로 합칩니다. 취합 한 건에 대략 15분 넘게 걸립니다. 판단은 거의 없고 규칙은 명확한 — 자동화 이득이 가장 큰 전형적인 반복 업무입니다.”

팁

멘토가 “실제 업무냐”고 물으면 — 시스템 개선요청 취합처럼 부서 취합이 필요한 실무 시나리오라고 답하세요.

03한 장 요약 · 처리 흐름≈ 40초
화면

메일 → 추출 → 업로드 → 5종 검증 → 병합 → 오류 보고서 → 요약 흐름.

대본

“전체 흐름을 한 장으로 보면 이렇습니다. 취합 요청 메일 한 건이 들어오면 시스템이 작성 항목과 제출 기준을 뽑아냅니다. 그다음 부서들이 제출한 엑셀 여러 개를 업로드하면, 5가지 규칙으로 검증하고, 정상 행만 병합하고, 오류는 보고서로 만들고, 마지막에 요약을 냅니다. 입력은 지저분하지만 출력은 깨끗합니다. 그리고 원본은 절대 건드리지 않고 결과는 전부 별도 파일로 만듭니다.”

팁

‘원본 보존 · 결과는 별도 파일’은 실무 신뢰 포인트라 꼭 한 번 언급.

04핵심 설계 철학≈ 60초 (핵심)
화면

LLM이 잘하는 일 vs 규칙이 잘하는 일 (2단) · LangGraph = 반장.

대본

“이 프로젝트에서 제가 가장 신경 쓴 설계 결정입니다. 저는 두 종류의 일을 분리했습니다. 왼쪽, LLM이 잘하는 일은 자연어 이해입니다 — 사람이 쓴 메일을 읽고 작성 항목과 마감을 뽑아내고, 요청 메일 초안을 쓰는 것. 이건 조금 틀려도 되고 창의적이어도 되는 영역입니다. 오른쪽, 규칙이 잘하는 일은 엑셀 검증입니다 — 이건 항상 똑같아야 하고 재현되어야 하는 영역이라 pandas 같은 코드로 처리했습니다. 그리고 LangGraph는 이 나뉜 에이전트들이 어떤 순서로 협업할지 정해 주는 ‘반장’입니다.”

팁

여기서 시간을 가장 많이 쓰세요. 멘토가 가장 관심 가질 슬라이드. ‘분리’를 손짓으로 좌/우 나눠 설명하면 각인됨.

05시스템 아키텍처≈ 45초
화면

Frontend → API → Orchestration → Engine 4계층 · 설계 경계.

대본

“구현은 3계층입니다. 프론트엔드는 React와 TypeScript, API 서버는 FastAPI, 그 뒤에 LangGraph 멀티에이전트 그래프가 있고, 실제 엑셀 처리는 pandas와 openpyxl 규칙 엔진이 합니다. 메일 이해에는 Azure OpenAI를 쓰지만 키가 없어도 휴리스틱으로 동작하게 만들었습니다. 여기서 중요한 경계는 — LLM은 메일 이해와 초안에만 쓰고, 검증과 병합은 결정론 코드가 한다는 것입니다.”

팁

“키 없어도 동작”은 데모 안정성 어필 포인트. 발표장 인터넷/키 문제로 안 멈춘다는 뜻.

06멀티에이전트 흐름 (LangGraph)≈ 55초
화면

START→Requirement→Supervisor→[RAG?]→Validation→Self-Correction→Merge/Report→END 타임라인. 파란=LLM, 초록=규칙.

대본

“LangGraph 내부에서 에이전트들이 순서대로 협업합니다. 먼저 Requirement Analysis가 메일을 분석하고, Supervisor가 검증 전략을 세우면서 ToT라는 기법으로 검증 규칙을 확정합니다. 필요하면 RAG로 기준 문서를 참고하고, Excel Validation이 5종 규칙 검증을 합니다. 그다음 Self-Correction이 안전하게 고칠 수 있는 오류만 고치고, 마지막으로 병합·오류 보고서·요약을 만듭니다. 화면에서 파란 태그가 LLM을 쓰는 단계, 초록 태그가 규칙만 쓰는 단계입니다 — 보시면 검증과 병합은 전부 규칙입니다.”

팁

태그 색을 손으로 가리키며 ‘LLM vs 규칙’을 시각적으로 다시 강조. 4번 슬라이드 메시지의 반복 확인.

07검증 규칙 5종≈ 45초
화면

필수값 / 필수 컬럼 / 날짜 형식 / 코드값 / 중복 + 원칙 칩 3개.

대본

“검증 규칙은 5가지입니다. 필수값이 비었는지, 필수 컬럼 자체가 없는지, 날짜가 YYYY-MM-DD 형식인지, 긴급도가 상·중·하 중 하나인지, 그리고 파일 경계를 넘어 중복이 있는지. 세 가지 원칙이 있습니다 — 결정론적이라 같은 입력이면 항상 같은 결과, 원본은 보존, 그리고 오류는 자동으로 고치지 않고 보고서에 남깁니다. 마음대로 고치면 신뢰를 잃으니까요.”

팁

중복은 “파일 하나 안이 아니라 여러 파일을 합쳐서 본다”는 점이 포인트.

08에이전틱 기법 3가지≈ 50초
화면

ToT · Self-Refine(Self-Correction) · 휴리스틱 폴백.

대본

“조금 더 기술적인 부분입니다. 세 가지 에이전틱 기법을 썼습니다. 첫째 ToT — 검증 규칙 후보를 세 개 만들어 실제 업로드된 엑셀 헤더로 채점해 가장 맞는 걸 고릅니다. 둘째 Self-Refine — LLM이 교정값을 제안하면 코드가 그걸 검증해 오류가 실제로 줄 때만 받아들입니다. 셋째 폴백 — Azure 키가 없어도 전 기능이 동작합니다. 공통점은, LLM은 ‘제안’만 하고 최종 ‘확정’은 항상 코드가 한다는 겁니다.”

팁

용어(ToT·Self-Refine)를 모르는 멘토에겐 “규칙 후보 채점”, “고쳐보고 나빠지면 되돌림”으로 풀어 설명.

09왜 검증을 LLM에 안 맡겼나≈ 45초
화면

LLM 검증=불안정 vs 규칙 검증=재현 (2단) + 논지 인용.

대본

“멘토님이 ‘왜 검증까지 LLM한테 안 맡겼냐’고 물으실 수 있어 따로 준비했습니다. LLM으로 검증하면 같은 파일인데도 실행마다 행 번호나 오류 유형이 흔들릴 수 있고 비용도 듭니다. 규칙으로 하면 같은 입력에 항상 같은 결과가 나와서, 발표나 감사에서 근거를 그대로 재현할 수 있습니다. 검증은 창의적인 답이 아니라 정확하고 재현 가능한 답이 필요하기 때문입니다.”

팁

이 슬라이드가 곧 예상 질문의 선제 답변. 멘토가 이미 물었다면 “다음 슬라이드에 답이 있다”며 넘어가도 좋음.

10실측 벤치마크≈ 55초
화면

F1 100% · 재현성 표준편차 0 · 46행 0.057초 · 99.99%↓ · Self-Correction 100% · ToT 오탐 1건 회피.

대본

“숫자로 검증한 결과입니다. 라벨을 붙인 데이터셋 4개 파일, 오류 10건으로 측정했습니다. 검출 F1은 100% — 10개 전부 잡고 오탐이 0입니다. 열 번 반복해도 결과 표준편차가 0, 즉 완벽히 재현됩니다. 46행을 0.057초에 처리하는데 사람이 하면 15분 넘게 걸리니 99.99% 단축입니다. Self-Correction은 고칠 수 있는 오류 6건을 전부 안전하게 고쳤고, ToT는 스키마가 바뀐 상황에서 오탐 1건을 회피했습니다.”

팁

숫자 다 읽지 말고 F1 100%와 표준편차 0 두 개만 확실히 강조. 근거 파일은 data/benchmark_metrics.json.

11정직한 스코프 (1차 PoC)≈ 55초
화면

되는 것(확장 포함) vs 방식 명확화·후속 (2단) + 원칙 유지 라인.

대본

“스코프를 정직하게 구분한 부분입니다. 처음엔 Gmail을 앱에서 자동화하지 않는다고 뒀는데, 이번에 확장(Phase A/B)으로 실제로 만들었습니다. 지금 되는 건 — 기존 5종 검증·병합·보고에 더해, 수신함을 분류하고 담당자별 요청서 초안을 만들고, 고급 RAG로 근거를 검증해 검토 큐에 올린 뒤 사람이 승인해 발송하는 것까지입니다. 다만 방식을 분명히 합니다 — 실제 Gmail 읽기·발송은 credentials를 넣으면 동작하고 기본은 mock이라 안전하며, 하루 1회 스케줄러는 후속이라 지금은 수집 버튼으로 돌립니다. 대신 원칙은 그대로입니다 — 자동 발송 안 함(사람 승인), 담당자는 조직도, 근거 없으면 플래그, 검증은 결정론.”

팁

핵심 프레이밍 — “‘안 한다’가 ‘했다’로 바뀌었지만 Human-in-the-loop와 근거 표시는 유지”. 약점이 아니라 성숙한 확장으로 설명. (이전 발표의 “Gmail 자동화 안 함”과 모순되지 않게 이렇게 말하세요.)

12확장 Phase A · Gmail 파이프라인≈ 55초
화면

수신함 수집→분류→요청서 초안→검토 큐→승인 발송 흐름 + 실제 검토 큐 화면.

대본

“확장의 첫 부분, Gmail 파이프라인입니다. 수신함을 수집하고(기본 mock, credentials 넣으면 실제 gmail.readonly), 일반과 취합요청을 확신도와 함께 분류합니다. 취합요청이면 담당자별 요청서 초안을 만드는데, 이때 수신자는 RAG로 추측하지 않고 조직도에서만 가져옵니다. 그리고 근거 검증 결과와 함께 검토 큐에 올리고, 사람이 승인해야 발송됩니다. 오른쪽 실제 화면을 보면 취합요청 두 건은 초안이 생성돼 승인·발송 버튼이 있고, 확신도 중간은 확인 필요로, 회식·뉴스레터는 일반으로 분류됐습니다. LLM은 이해·분류·작성, 발송 결정은 사람입니다.”

팁

‘조직도에서만 수신자’와 ‘자동 발송 안 함’ 두 가지를 꼭 강조 — 실무 안전장치.

13확장 Phase B · 고급 RAG≈ 50초
화면

메타데이터 필터링 · 쿼리 재작성 · 재정렬 · 근거 검증 (4카드) + 핵심 라인.

대본

“확장의 두 번째, 고급 RAG입니다. 네 가지를 넣었습니다 — 문서 태그로 검색 범위를 좁히는 메타데이터 필터링, 메일을 검색에 적합한 질의로 바꾸는 쿼리 재작성, 관련도로 상위 문서만 고르는 재정렬, 그리고 가장 중요한 근거 검증입니다. 생성될 요청 메일의 작성 항목·기한·담당자·규칙 근거가 실제 자료에 근거하는지 확인하고, 없으면 ‘확인 필요’로 플래그합니다. 앞 화면에서 한 건은 근거 100%, 다른 건은 75%에 ‘작성 항목’ 플래그가 뜬 게 이겁니다. 핵심은 RAG를 많이 쓰는 게 아니라 AI가 임의로 지어내지 않도록 근거를 검증하는 것이고, 임베딩 없이 구현했습니다.”

팁

근거 100% vs 75%+플래그 대비가 이 프로젝트 논지(‘LLM 제안, 코드 검증’)의 결정적 예시.

14그 외 개선 · 하드 샘플≈ 40초
화면

하드 샘플 생성기(오류 5종·스키마 드리프트·통화 숫자·파일 간 중복) + 테스트 87 통과.

대본

예상 질문 & 답

Q 왜 규칙 검증?  A 재현성. 같은 입력 → 같은 행·같은 오류.

Q Gmail을 실제로 읽나?  A 기본 mock, credentials 설정 시 실제 gmail.readonly. 분류·초안까지만, 자동 발송 안 함(사람 승인).

Q 담당자 주소는?  A RAG로 추측하지 않고 조직도(DB)에서만. 잘못된 발송 방지.

Q RAG는?  A 메타데이터 필터·쿼리 재작성·근거 검증까지 하는 고급 RAG. 임베딩 없이, 근거 없으면 플래그.

Q Self-Correction은 다 고치나?  A 아니오. 날짜·코드값만 안전 교정, 필수값·중복은 재제출 대상으로 남김.
