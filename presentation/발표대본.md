# Smart Collect Assistant · 슬라이드별 발표 대본

발표자 이형진(08079) · 15슬라이드 · 총 ≈ 13–14분
덱: https://deck-snowy-two.vercel.app · GitHub: github.com/PinokioLee/smart-collect-assistant

## 발표 전 30초 — 핵심 메시지 3가지
1. LLM(메일 이해·분류·작성)과 규칙(엑셀 검증)을 **분리**했다.
2. 검증은 **결정론적** — 같은 입력이면 항상 같은 결과 (F1 100%, 재현성 표준편차 0).
3. Gmail 수신함 분류·근거 검증까지 **확장**하되, 발송은 사람이 승인(Human-in-the-loop).

시연 준비물: 덱 URL(휴대폰) · GitHub · (선택) `run --hard --graph` 출력 캡처.

**팁**: 시간을 가장 많이 쓰는 슬라이드. 손짓으로 좌/우 나눠 설명.

### 11 · 정직한 스코프 (≈55초)
**화면**: 되는 것(확장 포함) vs 방식 명확화·후속 + 원칙 유지.
**대본**: "처음엔 Gmail을 앱에서 자동화하지 않는다고 뒀는데, 이번에 확장(Phase A/B)으로 실제로 만들었습니다. 지금 되는 건 — 5종 검증·병합·보고에 더해, 수신함 분류·담당자별 요청서 초안·근거 검증·검토 큐·사람 승인 후 발송까지입니다. 다만 방식을 분명히 — 실제 Gmail 읽기·발송은 credentials 넣으면 동작하고 기본은 mock이라 안전하며, 하루 1회 스케줄러는 후속이라 지금은 수집 버튼입니다. 원칙은 그대로 — 자동 발송 안 함(사람 승인), 담당자=조직도, 근거 없으면 플래그, 검증은 결정론."

### 12 · 확장 Phase A · Gmail 파이프라인 (≈55초)
**화면**: 수집→분류→요청서 초안→검토 큐→승인 발송 + 실제 검토 큐 화면.
**대본**: "수신함을 수집하고(기본 mock, credentials 넣으면 실제 gmail.readonly), 일반/취합요청을 확신도와 함께 분류합니다. 취합요청이면 담당자별 요청서 초안을 만드는데, 수신자는 RAG로 추측하지 않고 조직도에서만 가져옵니다. 근거 검증과 함께 검토 큐에 올리고, 사람이 승인해야 발송됩니다. 실제 화면을 보면 취합요청 2건은 초안 생성, 확신도 중간은 확인 필요, 회식·뉴스레터는 일반으로 분류됐습니다."

### 13 · 확장 Phase B · 고급 RAG (≈50초)
**화면**: 메타데이터 필터링·쿼리 재작성·재정렬·근거 검증(4카드) + 핵심 라인.
**대본**: "고급 RAG 네 가지 — 문서 태그로 범위를 좁히는 메타데이터 필터링, 검색용 질의로 바꾸는 쿼리 재작성, 상위 문서만 고르는 재정렬, 그리고 가장 중요한 근거 검증. 생성될 요청 메일의 작성 항목·기한·담당자·규칙 근거가 실제 자료에 근거하는지 확인하고, 없으면 확인 필요로 플래그합니다. 앞 화면에서 한 건은 근거 100%, 다른 건은 75%에 ‘작성 항목’ 플래그가 뜬 게 이겁니다. 핵심은 근거를 검증해 임의 생성을 막는 것이고, 임베딩 없이 구현했습니다."
**팁**: 근거 100% vs 75%+플래그 대비가 ‘LLM 제안, 코드 검증’ 논지의 결정적 예시.

### 15 · 예상 Q&A · 데모 · 마무리 (≈50초)
**화면**: 예상 질문 4개 + 데모 명령 + 마무리 한 줄.
**대본**: "예상 질문 — 왜 규칙? 재현성. Gmail 실제로 읽냐? 기본 mock, credentials 넣으면 실제, 자동 발송은 안 하고 사람 승인. 담당자 주소는? 조직도에서. RAG는? 고급 RAG. 데모는 `run --hard --graph`와 `inbox`, UI는 05 수신함 레인까지. 정리하면 — AI의 이해력과 코드의 정확성을 분리해 반복 취합 업무를 재현 가능하게 자동화했습니다. 감사합니다."

**예상 질문 & 답**
- Q 왜 규칙 검증? / A 재현성. 같은 입력 → 같은 행·오류.
- Q Gmail을 실제로 읽나? / A 기본 mock, credentials 설정 시 실제 gmail.readonly. 분류·초안까지만, 자동 발송 안 함(사람 승인).
- Q 담당자 주소는? / A RAG로 추측 안 하고 조직도(DB)에서만. 잘못된 발송 방지.
- Q RAG는? / A 메타데이터 필터·쿼리 재작성·근거 검증까지 하는 고급 RAG. 임베딩 없이, 근거 없으면 플래그.
- Q Self-Correction 다 고치나? / A 아니오. 날짜·코드값만 안전 교정, 필수값·중복은 재제출 대상.
- Q 왜 Vercel 정적 배포? / A 슬라이드는 정적이라 Vercel에. 백엔드는 파일 저장·무거운 의존성 때문에 상태 유지형 호스트 필요 → 로컬/후속.
