# Smart Collect 시연 영상 녹음 대본

## 시연 개요

- 목표 시간: 5분 이내
- 예상 시간: 약 4분 55초
- 녹화 방식: 화면 조작과 발표자 음성 녹음
- 자막: 없음
- 녹화용 접속 주소: `http://127.0.0.1:5173/?presentation=1`
- 시연 환경: Gmail API 기반 기능을 Mock 메일로 재현
- 핵심 사례
  1. 정상 제출 및 최종 취합 완료
  2. 작업번호 없는 제출의 안전한 예외 처리

### 시연 대상 메일 및 Langfuse 로그

| 사례 | 이벤트 ID | 제목 | 발신자 | 첨부파일 | 최신 Langfuse 시각 |
|---|---|---|---|---|---|
| CASE 1 | `DEMO-GAMMA-OK` | `[SC-DEMO-SALES] 실적 제출합니다` | `gamma@company.com` | `7월_프로젝트_실적_gamma.xlsx` | `05:49:13`~`05:49:18` |
| CASE 2 | `DEMO-ORPHAN` | `취합 파일 제출합니다` | `unknown@company.com` | `corrected_submission.xlsx` | `05:48:44`~`05:48:47` |

> Langfuse 시간은 2026년 7월 23일 화면 표시 시각(KST) 기준이다. 같은 이벤트 ID로 반복 시연한 Trace가 누적되어 있으므로, 이름만 보고 고르지 말고 아래에 적힌 시각의 행을 선택한다.

## 0:00~0:25 — 시연 환경 설명

### 화면 조작

1. Smart Collect의 초기 화면을 보여준다.
2. `전체` 필터가 선택되어 있고 메일 목록이 비어 있는 상태를 보여준다.

### 발표 대본

> 지금부터 사내 자료 취합 업무를 자동화하는 프로그램을 시연하겠습니다.
>
> 사내 메일은 보안상 사용할 수 없어 Gmail API를 이용해 메일을 가져오도록 구현했습니다.
>
> 운영 환경에서는 스케줄러가 정해진 시간마다 메일을 자동으로 가져오지만, 오늘은 시연을 위해 수동으로 실행하겠습니다.

## 0:25~1:20 — 메일 수신 및 Agent 자동 처리

### 화면 조작

1. `지금 메일 확인` 버튼을 클릭한다.
2. 처리가 끝날 때까지 기다린다.
3. 상단 처리 현황과 메일 목록을 보여준다.
4. `전체 / 처리 완료 / 확인 필요 / 격리` 필터를 차례로 짧게 보여준다.

### 발표 대본

> 현재는 처리한 메일이 없는 초기 상태입니다.
>
> 지금 메일 확인 버튼을 누르면 Inbox Intake Agent가 메일을 분석해 업무 유형을 분류하고, Supervisor Agent가 다음 처리 경로를 결정합니다.
>
> 작성 양식이 첨부되어 있으면 그대로 활용하고, 없으면 메일에서 추출한 항목으로 새로운 엑셀 양식과 검증 규칙을 생성합니다.
>
> 자료 제출은 Validation Agent가 검증하고 병합하며, 불확실한 메일은 사람에게 전달하고 위험 메일은 격리합니다.
>
> 핵심은 LLM이 업무를 판단하고, 코드는 데이터와 실행 조건을 검증하며, 코드가 확정할 수 없는 예외만 사람에게 전달한다는 점입니다.
>
> 처리 결과는 전체, 처리 완료, 확인 필요, 격리로 나누어 볼 수 있습니다. 오늘은 정상 완료와 안전한 예외 처리, 두 가지 사례를 보여드리겠습니다.

# CASE 1. 정상 제출 및 최종 취합 완료

## 1:20~2:20 — 앱 처리 결과

### 화면 조작

1. `처리 완료` 필터를 클릭한다.
2. 아래 메일 카드를 선택한다.
   - 제목: `[SC-DEMO-SALES] 실적 제출합니다`
   - 발신자: `gamma@company.com`
   - 이벤트 ID: `DEMO-GAMMA-OK`
   - 수신 첨부파일: `7월_프로젝트_실적_gamma.xlsx`
3. 첨부파일명과 판단 근거를 보여준다.
4. `판단 근거와 실행 로그`를 펼친다.
5. `Supervisor Agent`, `Validation Contract Agent`, `Self-Correction Agent`, `Final Validation Agent` 순서를 가리킨다.
6. `전체` 필터로 전환한다.
7. 같은 메일의 최종 취합 완료 결과에서 병합 파일 `SC-DEMO-SALES_merged.xlsx`와 완료 메일 초안을 보여준다.

### 발표 대본

> 첫 번째는 정상 제출이 자동 처리되는 과정입니다.
>
> Inbox Intake Agent가 제출 메일로 분류하면 Supervisor Agent가 Validation Agent로 전달합니다. Validation Agent는 작업번호와 첨부파일을 확인한 뒤 필수 컬럼과 데이터 형식을 검증합니다.
>
> 이 실행 로그는 SQLite에 저장된 처리 기록을 화면에서 읽기 쉽게 보여주는 것입니다.
>
> 메일 분류와 Agent 선택, 완료 메일 작성은 LLM이 담당합니다. 반면 검증 규칙 적용, 교정 결과 재검증, 최종 승인과 엑셀 병합은 코드가 담당합니다.
>
> LLM이 교정값을 제안해도 코드가 허용값과 형식을 다시 검사하며, 검증을 통과한 데이터만 취합 파일에 반영합니다. 메일 발송도 코드의 정책 Gate를 통과해야 합니다.
>
> 모든 제출이 완료되면 최종 취합 파일과 완료 메일 초안이 생성됩니다. 시연 환경에서는 실제 메일을 발송하지 않습니다.

## 2:20~3:35 — Langfuse 확인

### 화면 조작

1. Langfuse로 전환한다.
2. 검색창에 `name:inbox_classification`을 입력한다.
3. `2026-07-23 05:49:13` 행을 클릭한다.
4. 열린 Trace 제목이 `inbox_classification: DEMO-GAMMA-OK:agentic`인지 확인한다.
5. 오른쪽 `User` 영역에서 다음 메일 정보가 보이도록 한다.
   - 제목: `[SC-DEMO-SALES] 실적 제출합니다`
   - 보낸 사람: `gamma@company.com`
   - 첨부: `7월_프로젝트_실적_gamma.xlsx`
6. `Assistant` 영역의 `category=collection`, `intent=submission`, `confidence=0.98`, `reason`을 가리킨다.
7. 상단의 모델명 `gpt-4.1`, 토큰, 비용과 Metadata의 `latency_ms`를 보여준다.
8. Trace Tree에서 현재 실행 묶음의 `supervisor_route`를 선택한다. 반복 항목이 헷갈리면 표로 돌아가 `name:supervisor_route`를 검색하고 `05:49:16` 행을 선택한다.
9. `action=submission_worker`와 선택 근거를 보여준다.
10. Trace Tree에서 현재 실행 묶음의 `completion_report_mail`을 선택한다. 반복 항목이 헷갈리면 `name:completion_report_mail`을 검색하고 `05:49:18` 행을 선택한다.
11. 최종 완료 메일 생성 결과를 짧게 보여준다.

### 발표 대본

> Langfuse는 메일 한 건을 처리하면서 발생한 LLM 호출을 Trace로 기록합니다.
>
> 왼쪽 Tree에는 호출 순서가 표시됩니다. inbox classification은 메일 분류, supervisor route는 다음 Worker를 선택한 기록입니다.
>
> 오른쪽 User 영역에는 LLM에 전달된 메일과 지시가, Assistant 영역에는 category, intent, 신뢰도와 판단 사유가 구조화되어 표시됩니다.
>
> supervisor route에서는 Validation Agent를 선택한 action과 근거를 확인하고, completion report mail에서는 최종 메일 생성 결과를 확인할 수 있습니다.
>
> 상단에는 모델과 토큰, 비용이 표시되고 Metadata에는 처리시간이 기록됩니다.
>
> Langfuse는 Agent 실행 과정을 관찰하는 도구입니다. 오픈소스와 Self-host 선택지가 있고, 입력·출력과 비용·시간을 한 화면에서 확인할 수 있어 채택했습니다.

# CASE 2. 작업번호 없는 제출의 안전한 예외 처리

## 3:35~4:20 — 앱 처리 결과

### 화면 조작

1. 앱으로 돌아간다.
2. `확인 필요` 필터를 클릭한다.
3. 아래 메일 카드를 선택한다.
   - 제목: `취합 파일 제출합니다`
   - 발신자: `unknown@company.com`
   - 이벤트 ID: `DEMO-ORPHAN`
   - 첨부파일: `corrected_submission.xlsx`
4. 원본 메일에 작업번호가 없다는 내용과 `job_not_found` 사유를 보여준다.
5. `판단 근거와 실행 로그`를 펼친다.
6. Validation Agent 실패, Supervisor 재시도, 사람 검토 전환 순서를 보여준다.

### 발표 대본

> 두 번째는 제출 파일은 있지만 작업번호가 없는 경우입니다.
>
> Intake Agent는 제출 메일로 분류하지만, Validation Agent가 연결할 작업을 찾지 못하면 `job_not_found`를 반환합니다.
>
> Supervisor Agent는 실패 원인을 관찰하고 다시 시도합니다. 그래도 안전하게 연결할 수 없으면 임의로 처리하지 않고 사람의 확인이 필요한 상태로 전환합니다.
>
> 즉, LLM이 경로를 판단하고 코드가 실제 연결 가능 여부를 검증한 뒤, 코드로 확정할 수 없는 예외만 사람에게 전달합니다.

## 4:20~4:35 — Langfuse 확인

### 화면 조작

1. Langfuse로 전환한다.
2. 검색창에 `name:inbox_classification`을 입력한다.
3. `2026-07-23 05:48:44` 행을 클릭한다.
4. 열린 Trace 제목이 `inbox_classification: DEMO-ORPHAN:agentic`인지 확인한다.
5. `User` 영역에서 제목 `취합 파일 제출합니다`, 발신자 `unknown@company.com`, 첨부 `corrected_submission.xlsx`를 보여준다.
6. `Assistant` 영역의 `category=collection`, `intent=submission`, `confidence=0.95`를 보여준다.
7. 표로 돌아가 검색어를 `name:supervisor_route`로 변경한다.
8. `2026-07-23 05:48:47` 행을 클릭한다.
9. Trace가 `DEMO-ORPHAN:agentic`인지 확인하고 `action=submission_worker`와 판단 근거를 보여준다.

### 발표 대본

> Langfuse에서 제출 메일 분류와 Validation Agent를 선택한 근거를 확인할 수 있습니다.
>
> 이후 작업번호를 찾지 못한 결과와 사람 검토 전환 과정은 앱의 실행 로그에 기록됩니다.

## 녹화 팁

- 화면을 클릭한 직후 말하지 말고, 결과가 나타난 다음 설명한다.
- 마우스를 빠르게 흔들지 말고 다음에 보여줄 위치로 천천히 이동한다.
- Langfuse에서는 입력 전체를 읽지 말고 `category`, `action`, `reason`만 보여준다.
- Langfuse Trace Tree에는 반복 시연 기록이 누적되어 있다. 다른 실행을 잘못 선택하지 않도록 CASE 1은 `05:49:13 / 05:49:16 / 05:49:18`, CASE 2는 `05:48:44 / 05:48:47` 시각을 기준으로 찾는다.
- 현재 앱의 Mock 데이터 처리 시각은 `07:54`이고 Langfuse의 최신 일치 LLM 로그는 `05:48~05:49`이다. `prepare_agentic_demo.py`는 로컬 시연 데이터를 준비하지만 새로운 Langfuse 호출을 만들지는 않는다.
- 앱의 `판단 근거와 실행 로그`가 SQLite 기록을 보여주므로 PowerShell 화면은 열지 않는다.
- 실제 발송이 비활성화된 시연 환경이라는 점은 한 번만 설명한다.
- 녹화 전 알림, TV, 메신저 소리를 끄고 마이크 입력 레벨을 확인한다.
