Smart Collect Assistant 시스템 아키텍처

취합 요청 메일 분석, 작성 가이드 생성, 제출 현황 추적, 엑셀 병합 및 검증을 상태 기반 Agentic AI 워크플로로 설계한다.

1. 아키텍처 판단 요약

문제 성격 여러 작성자, 메일, 파일, 마감, 검증 규칙이 하나의 취합 케이스 안에서 계속 상태를 바꾼다. 단순 챗봇보다 상태 저장 워크플로가 적합하다.
PoC 우선 범위 엑셀 병합/검증, 제출 현황 추적, AI 기반 요청 분석/가이드 생성을 우선 구현한다. SMS와 완전 자동 발송은 승인 기반 후순위로 둔다.
핵심 설계 원칙 LLM은 해석, 분해, 초안 생성, 라우팅에 집중한다. 엑셀 병합과 검증은 결정론적 도구로 실행해 재현성과 감사 가능성을 확보한다.

2. 선택한 Agent 디자인 패턴

패턴 적용 위치 선택 이유
Magnetic Sticky Router 취합 케이스 전체 오케스트레이션 메일 분석, 제출 추적, 검증, 리마인드, 보고가 상태에 따라 반복 분기되므로 중앙 라우터가 최신 State를 보고 다음 전문 에이전트를 선택한다.
Router & Specialist 메일 분석, 가이드 생성, 제출 추적, 엑셀 처리, 보고서 생성 한 거대 에이전트에 모든 도구를 주지 않고, 역할별 전문 에이전트와 최소 권한 도구로 나눠 품질과 보안을 높인다.
Planning + ReAct 요청 메일 해석, 누락 정보 확인, 제출/검증 후속 조치 취합 요청에서 목적, 항목, 기한, 대상자, 첨부 양식을 분리하고 필요한 도구를 순차 호출해야 한다.
Reflection 작성 가이드/메일 초안 품질 검토, 검증 결과 요약 검토 작성자가 이해하기 쉬운 안내문과 오류 보고서는 품질 기준이 중요하므로 초안 생성 후 비평/개선 루프를 둔다.
Tool Use + MCP Gateway Outlook/메일, 파일 저장소, Excel 파서, 알림, 보고서 생성 여러 내부 시스템을 에이전트 안전 함수로 노출해야 하므로 단일 MCP 게이트웨이로 도구 접근을 통제한다.
Human-in-the-Loop 메일/SMS 발송, 미제출 리마인드, 최종 파일 확정 오발송, 개인정보, 잘못된 최종 취합을 막기 위해 위험 행동은 담당자 승인 후 실행한다.

결론: 이 시스템은 "순차적 핸드오프"만으로도 PoC를 만들 수 있지만, 실제 업무 흐름은 마감 상태와 오류 상태에 따라 되돌아가야 한다. 따라서 중앙 라우터가 전체 State를 읽고 전문 에이전트를 호출하는 구조가 더 적합하다.

4. 핵심 State 모델

CollectionCase

  • case_id, title, owner, deadline
  • request_spec: 목적, 작성 항목, 제출 방법, 주의사항
  • recipients: 이름, 부서, 이메일, 연락처, 제출 상태
  • guide_draft, mail_draft, approval_status

Execution Evidence

  • submissions: 파일명, 제출자, 수신 시간, 버전
  • validation_results: 누락, 형식 오류, 중복 의심, 정상 건수
  • final_artifacts: 취합본, 오류 보고서, 제출 현황 보고서
  • audit_trail: 도구 호출, 승인자, 판단 근거, 실패/재시도

7. 에이전트별 책임과 도구 권한

에이전트 책임 허용 도구 금지/제한
Collection Orchestrator State 기반 라우팅, 작업 분해, 완료 조건 판단, 재시도 제어 State DB, Agent 호출, Audit Log 직접 메일 발송/파일 수정 금지
Request Analysis Agent 취합 요청 목적, 작성 항목, 기한, 제출 방법, 대상 추출 메일 읽기, 첨부 메타데이터 읽기, 템플릿 검색 발송/삭제/수정 권한 없음
Guide & Mail Draft Agent 작성자용 쉬운 가이드와 메일 초안 생성 템플릿 검색, 초안 저장 승인 없는 발송 금지
Submission Tracking Agent 회신 메일과 업로드 이벤트 감지, 제출 상태 갱신, 미제출자 산출 메일 읽기, 파일 메타데이터 읽기, 상태 업데이트 리마인드 자동 발송은 승인 필요
Excel Merge & Validation Agent 표준 컬럼 매핑, 병합 실행, 필수값/형식/중복 검증 Excel 파서, 파일 읽기/쓰기, 검증 규칙 업무 내용 임의 수정 금지, 오류는 보고/요청만 수행
Report Agent 제출률, 오류 건수, 최종 산출물 링크, 후속 액션 요약 State 읽기, 보고서 생성, 메일 초안 저장 완료 보고 자동 발송은 승인 필요

8. 배포 및 통합 구조

PoC 구조

  • 웹 UI: 취합 케이스 등록, 승인, 검증 결과 확인
  • Backend API: 케이스/수신자/제출/검증 결과 관리
  • Agent Runtime: LangGraph류 State-Node-Edge 워크플로
  • Excel Tool: pandas/openpyxl 기반 결정론적 병합/검증
  • 메일 연동: 초기에는 읽기/초안 생성 중심, 발송은 승인 후

운영 확장 구조

  • MCP Gateway: 메일, 파일 저장소, 알림, 보고서 도구의 단일 진입점
  • AgentOps: 추론 단계, 도구 호출, 토큰 비용, 실패율, 승인 이력 추적
  • 권한 스코프: 읽기, 초안 작성, 발송, 파일 생성 권한 분리
  • 버전 관리: 프롬프트, 검증 규칙, MCP manifest, Excel 템플릿 버전 고정

9. 보안, 품질, 실패 대응

오발송 방지

메일/SMS/리마인드는 모두 담당자 승인 후 실행한다. 수신자, 제목, 본문, 첨부 파일을 승인 카드에서 확인하게 한다.

AI 판단 한계

LLM은 작성 기준을 제안하고 설명하지만, 실제 데이터 수정과 최종 확정은 담당자와 결정론적 검증 도구가 담당한다.

감사 가능성

모든 도구 호출, 승인, 검증 결과, 생성 파일 버전을 케이스 감사 로그에 남겨 결과 신뢰도를 설명할 수 있게 한다.
실패 상황 대응
메일 분석 신뢰도 낮음 추출 결과를 확정하지 않고 담당자에게 누락 항목 확인을 요청한다.
Excel 양식이 예상과 다름 컬럼 매핑 후보를 제시하고, 담당자 승인 후 병합한다. 매크로/복잡 양식은 범위 밖으로 분류한다.
제출 파일 오류 발생 오류 행, 컬럼, 원인, 수정 요청 문구를 보고서로 생성하되 원본 파일은 수정하지 않는다.
에이전트 반복 루프 과다 최대 루프 수와 비용 한도를 두고 초과 시 담당자 검토 상태로 전환한다.
