## 1. 전체 구조 한눈에 보기

Smart Collect Assistant는 취합 요청 메일과 여러 부서의 엑셀 파일을 받아서, 작성 기준을 파악하고, 제출 파일을 검증하고, 정상 데이터만 병합하고, 오류 보고서를 만들어주는 멀티 에이전트 PoC입니다.

화면은 취합 업무 순서 그대로 4단계로 구성됩니다.

```text
[1] 취합 요청 메일 보내기  → 받은 요청 분석 + 내 발송 스타일(RAG) 반영한 요청 초안 + 양식 첨부·발송
[2] 제출 엑셀 검증·병합    → 회신 첨부 검증 → 정상만 병합 → 취합 엑셀 다운로드
[3] 제출 현황·리마인드     → 제출/미제출 파악 + 리마인드 메일 발송
[4] 공통 항목 일괄 수정    → 여러 파일 공통 컬럼 일괄 수정(원본 보존)
```

내부 처리 흐름은 다음과 같습니다.

```text
React 화면(4단계 파이프라인 UI)
  -> FastAPI 서버
    -> LangGraph Agent 흐름
      -> 메일 분석(Azure OpenAI / 휴리스틱)
      -> 검증 규칙 선택(Tree of Thoughts)
      -> 엑셀 검증(pandas/openpyxl 결정론)
      -> Self-Correction(안전한 오류만 정규화)
      -> 병합 파일 생성
      -> 오류 보고서 생성
      -> (전 노드) Langfuse 실행 추적
```

이 프로젝트에서 가장 중요한 설계 원칙은 다음입니다.

```text
LLM이 잘하는 자연어 이해와,
규칙 기반 코드가 잘하는 정확한 데이터 검증을 분리했다.
```

즉, AI에게 모든 것을 맡기지 않았습니다.

- 메일 분석 / 요청 메일 초안: Azure OpenAI 사용(키 없으면 휴리스틱 폴백)
- 내 발송 스타일 반영: 과거 발송 메일을 참고 문서로 검색하는 RAG(학습 아님)
- 검증 전략 계획: Supervisor(LLM)가 실제 컬럼을 보고 전략·리스크를 판단(계획만, 검증은 코드)
- 엑셀 검증·병합: pandas/openpyxl 기반 결정론적 규칙 사용(LLM 미사용)
- 자가교정: LLM이 교정값을 제안 → 코드가 허용값·재검증으로 확정(Propose–Verify)
- 실행 증거: 요청 1건마다 추론 트레이스 파일(`data/traces/*.md/.json`) 자동 기록
- 작업 흐름: LangGraph로 Agent 순서 관리
- 실행 추적: Langfuse 연동(멀티에이전트 실행 과정 시각화)
- Gmail 연동: 앱이 직접 읽지 않고 Claude Code의 Gmail MCP로 필요할 때만 가져오는 Human-in-the-loop
- 사용자 화면: React + TypeScript / API 서버: FastAPI

## 1.5 이것은 4개 기능이 아니라 하나의 에이전트 세션이다 (가장 중요)

### 자주 받는 지적

> "화면이 01~04로 나뉘어 있어 개별 기능처럼 보인다. LLM은 어디에 능동적으로 썼나?
> 그냥 Claude 웹에 메일 붙여넣어 요약받는 것과 뭐가 다른가?"

정당한 지적입니다. 그래서 핵심을 **‘단발 요약’이 아니라 ‘LLM이 판단하고 코드가 검증하는 에이전트 루프’**로 재설계했습니다.

### 설계 원칙: LLM은 판단(판단이 필요한 곳), 코드는 검증·확정(정확성이 필요한 곳)

취합 1건이 들어오면 여러 에이전트가 **하나의 세션**으로 이어집니다. 각 단계의 **판단 주체**가 다릅니다.

```text
[ANALYZE]  🧠 LLM   메일에서 작성항목·마감·주의사항 추출
[PLAN]     🧠 LLM   Supervisor가 실제 업로드 컬럼을 보고 '검증 전략'과 '리스크'를 판단
[BRANCH×3] ⚙️ 코드   Tree of Thoughts로 규칙 후보 3개 생성
[SELECT]   ⚙️ 코드   coverage 게이트로 규칙 결정론적 선택
[VALIDATE] ⚙️ 코드   4종 규칙 검증 (재현성 표준편차 0)
[CRITIQUE] ⚙️ 코드   교정 가능한 오류만 선별
[PROPOSE]  🧠 LLM   각 오류의 교정값을 '제안'(근거 포함)
[VERIFY]   ⚙️ 코드   제안값을 허용값·날짜형식으로 '검증' — 통과한 것만 채택(환각 차단)
[RE-VALIDATE]⚙️코드  재검증해 '오류가 줄 때만' 최종 채택
[MERGE/REPORT]⚙️코드 병합·오류보고서 생성
```

즉 **LLM에게 자연어 판단(계획·제안)을 맡기고, 코드가 정확한 검증·확정을 맡습니다.**
Claude 웹 단발 요약과의 결정적 차이는 세 가지입니다.

1. **LLM이 능동적으로 판단합니다.** Supervisor는 "이 파일들은 요청사유 컬럼이 없으니 balanced 전략,
   긴급도 코드값 흔들림에 주의"처럼 **전략과 리스크를 스스로 판단**합니다.
2. **LLM 제안을 코드가 검증합니다.** 자가교정에서 LLM이 `매우 급함→상`을 근거와 함께 제안하면,
   코드가 허용값 목록·날짜형식으로 **재검증**하고 재검증에서 **오류가 줄 때만** 채택합니다.
   그래서 LLM의 유연함을 쓰되 환각·비결정성은 결정론 게이트가 막습니다.
3. **과정이 증거로 남습니다.** 요청 1건마다 `data/traces/{요청ID}.md/.json`에
   "어느 에이전트가 / 어떤 국면에서 / LLM인지 규칙인지 / 무엇을 어떤 근거로 판단했는지"가 기록됩니다.

### 왜 검증까지 LLM에게 안 맡겼나 (일관성)

검증·병합은 여전히 결정론 코드입니다. LLM 직접 검증과 규칙 기반을 실측 비교하면 정확도는 F1 100%로
같지만 규칙 기반이 46행 기준 약 301배, 130행 기준 약 174배 빠르고 재현성 표준편차가 0입니다.
그래서 **판단이 필요한 곳(계획·제안)에만 LLM, 정확성·재현성이 필요한 곳(검증·확정)엔 코드**라는
원칙을 일관되게 지켰습니다.

### 멘토에게 설명할 말

```text
이 시스템은 메일을 한 번 요약하는 도구가 아니라,
LLM이 검증 전략을 '계획'하고 교정값을 '제안'하면 코드가 허용값·재검증으로 '확정'하는
Propose–Verify 에이전트 세션입니다.
그 판단 과정은 요청마다 트레이스 파일로 기록되어 '주장'이 아니라 '기록'으로 증명됩니다.
```

## 2. Python

### 무엇인가요?

Python은 컴퓨터에게 일을 시키는 프로그래밍 언어입니다. 엑셀 읽기, 파일 처리, 데이터 분석, AI 연동, API 서버 개발에 많이 사용됩니다.

초등학생식으로 말하면, Python은 "컴퓨터에게 순서대로 일을 시키는 말"입니다.

### 왜 썼나요?

이 과제는 엑셀 파일을 읽고, 행과 열을 검사하고, 오류 보고서를 다시 엑셀로 만들어야 했습니다. Python에는 이런 일을 잘하는 라이브러리인 `pandas`, `openpyxl`, `FastAPI`, `LangGraph`가 잘 준비되어 있습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

JavaScript만으로도 만들 수는 있지만, 엑셀 검증과 데이터 처리에서는 Python 생태계가 더 강합니다. Java나 C#도 가능하지만, PoC를 빠르게 만들기에는 코드와 설정이 무거워질 수 있습니다.

### 멘토에게 설명할 말

```text
엑셀 검증, 파일 처리, AI 연동을 빠르게 구현하기 위해 Python을 사용했습니다.
특히 pandas/openpyxl 생태계가 엑셀 자동화에 적합했습니다.
```

## 3. React

### 무엇인가요?

React는 웹 화면을 만드는 도구입니다. 사용자가 메일 내용을 넣고, 엑셀 파일을 올리고, 검증 결과를 확인하는 화면을 만들 때 사용했습니다.

초등학생식으로 말하면, React는 "버튼, 입력창, 결과표를 만드는 블록 장난감"입니다.

### 왜 썼나요?

취합 담당자가 브라우저에서 다음 작업을 해야 했습니다.

- 샘플 메일 불러오기
- 엑셀 파일 업로드
- 검증 실행
- 결과 요약 확인
- 병합 파일과 오류 보고서 다운로드

React는 이런 상호작용 화면을 만들기 좋습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

HTML만으로도 간단한 화면은 만들 수 있지만, 파일 업로드 상태나 API 결과를 다루기 불편합니다. Vue나 Svelte도 가능하지만 React는 가장 널리 쓰이고 자료가 많아 PoC에 안정적입니다.

### 멘토에게 설명할 말

```text
사용자가 직접 메일과 엑셀 파일을 넣고 결과를 확인해야 해서 React로 화면을 구성했습니다.
```

## 4. TypeScript

### 무엇인가요?

TypeScript는 JavaScript에 타입 검사를 더한 언어입니다. 예를 들어 숫자가 들어와야 하는 곳에 글자가 들어오면 미리 알려줍니다.

초등학생식으로 말하면, TypeScript는 "상자마다 이름표를 붙여서 잘못 넣는 실수를 줄여주는 도구"입니다.

### 왜 썼나요?

프론트엔드는 백엔드에서 검증 결과, 오류 목록, 다운로드 링크 같은 데이터를 받습니다. 이런 데이터 구조가 틀어지면 화면에서 오류가 나기 쉽습니다. TypeScript를 쓰면 실수를 미리 줄일 수 있습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

JavaScript만 쓰면 빠르게 만들 수는 있지만, 데이터 구조 실수를 늦게 발견할 수 있습니다. 이 과제는 결과 데이터 구조가 중요하기 때문에 TypeScript가 더 적합했습니다.

### 멘토에게 설명할 말

```text
API 응답 구조와 화면 상태를 안전하게 다루기 위해 TypeScript를 사용했습니다.
```

## 5. Vite

### 무엇인가요?

Vite는 React 앱을 빠르게 실행하고 빌드해주는 개발 도구입니다.

초등학생식으로 말하면, Vite는 "웹 화면을 빠르게 켜주는 시동 버튼"입니다.

### 왜 썼나요?

개발 서버가 빠르고 설정이 간단합니다. PoC에서 빠르게 화면을 만들고 확인하기 좋았습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

Create React App은 예전 방식이고 상대적으로 느립니다. Next.js는 좋은 프레임워크지만, 이 과제는 복잡한 웹사이트가 아니라 로컬 PoC 화면이므로 과합니다.

### 멘토에게 설명할 말

```text
React 개발과 빌드를 빠르게 하기 위해 Vite를 사용했습니다.
```

## 6. Axios

### 무엇인가요?

Axios는 프론트엔드에서 백엔드 API로 요청을 보내는 도구입니다.

초등학생식으로 말하면, Axios는 "화면에서 서버로 심부름을 보내는 배달부"입니다.

### 왜 썼나요?

React 화면에서 FastAPI 서버로 엑셀 파일을 업로드하고, 검증 실행 요청을 보내고, 결과를 받아오기 위해 사용했습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

브라우저 기본 `fetch`도 사용할 수 있습니다. 다만 Axios가 파일 업로드와 응답 처리 코드를 더 간단하게 만들어줍니다.

### 멘토에게 설명할 말

```text
프론트엔드와 백엔드 API 통신을 단순하게 만들기 위해 Axios를 사용했습니다.
```

## 7. FastAPI

### 무엇인가요?

FastAPI는 Python으로 API 서버를 만드는 도구입니다. React 화면과 Python 처리 로직 사이의 문 역할을 합니다.

초등학생식으로 말하면, FastAPI는 "화면에서 온 요청을 받아서 안쪽 일꾼에게 전달하는 접수처"입니다.

### 왜 썼나요?

이 프로젝트에는 다음 API가 필요했습니다.

- 서버 상태 확인
- 샘플 메일 조회
- 샘플 엑셀 생성
- 엑셀 파일 업로드
- 검증 및 병합 실행
- 결과 파일 다운로드

FastAPI는 파일 업로드, JSON 응답, 다운로드 API를 빠르게 만들 수 있습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

Flask도 가능하지만 타입 기반 API 작성과 문서화는 FastAPI가 더 편합니다. Django는 큰 서비스에는 좋지만, 이번 PoC에는 너무 무겁습니다.

### 멘토에게 설명할 말

```text
React 화면과 Python Agent 파이프라인을 연결하는 API 서버로 FastAPI를 사용했습니다.
파일 업로드와 결과 다운로드를 빠르게 구현할 수 있었습니다.
```

## 8. Uvicorn

### 무엇인가요?

Uvicorn은 FastAPI 앱을 실제로 실행해주는 서버 엔진입니다.

초등학생식으로 말하면, FastAPI가 가게라면 Uvicorn은 그 가게의 전기를 켜는 스위치입니다.

### 왜 썼나요?

FastAPI는 앱을 정의하는 도구이고, Uvicorn은 그 앱을 실행시키는 표준 실행기입니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

FastAPI와 가장 일반적으로 함께 쓰는 실행 서버가 Uvicorn입니다. 다른 서버도 가능하지만 PoC에서는 표준 조합을 쓰는 것이 안정적입니다.

### 멘토에게 설명할 말

```text
FastAPI 서버 실행을 위해 표준 ASGI 서버인 Uvicorn을 사용했습니다.
```

## 9. LangGraph

### 무엇인가요?

LangGraph는 여러 Agent의 작업 순서를 그래프처럼 연결해주는 도구입니다.

초등학생식으로 말하면, LangGraph는 "각 친구에게 순서대로 일을 시키는 반장"입니다.

이 프로젝트의 흐름은 다음과 같습니다.

```text
Requirement Analysis
  -> Planning
  -> Excel Validation
  -> Self-Correction
  -> Merge
  -> Error Report
  -> Report
```

### 왜 썼나요?

이 과제는 AI를 한 번 호출하는 단순 구조가 아닙니다. 메일 분석, 계획, 검증, 교정, 병합, 보고가 순서대로 이어집니다. LangGraph를 쓰면 각 단계를 Agent Node로 나누고, 어떤 순서로 실행되는지 명확하게 보여줄 수 있습니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

그냥 Python 함수만 순서대로 불러도 동작은 합니다. 하지만 그러면 Multi-Agent 구조가 잘 보이지 않습니다. LangGraph를 쓰면 발표에서 "각 Agent가 어떤 역할을 맡는지"를 설명하기 쉽습니다.

### 멘토에게 설명할 말

```text
단순 함수 호출이 아니라, Requirement Analysis, Planning, Validation, Self-Correction, Report로 나뉜 Agent 흐름을 명확히 보여주기 위해 LangGraph를 사용했습니다.
```

## 17. Tree of Thoughts, ToT

### 무엇인가요?

Tree of Thoughts는 답을 하나만 바로 고르지 않고 여러 후보를 만든 뒤 비교해서 선택하는 방식입니다.

초등학생식으로 말하면, ToT는 "답안 후보를 여러 개 써보고 제일 나은 답을 고르는 방식"입니다.

이 프로젝트에서는 검증 규칙 후보를 다음처럼 나눴습니다.

- Strict: 엄격하게 검사
- Balanced: 실제 양식과 기준 사이의 균형
- Loose: 최소한만 검사

### 왜 썼나요?

메일에서 추출된 작성 항목을 모두 필수 컬럼으로 강제하면 실제 양식이 조금 다를 때 오탐이 생길 수 있습니다. 예를 들어 메일에는 `요청사유`가 있지만 실제 엑셀 양식에는 없을 수 있습니다. 이때 무조건 오류로 보면 과잉 검증이 됩니다.

ToT 방식은 여러 규칙 후보를 만들고 실제 업로드 파일의 컬럼과 비교해서 더 적절한 규칙을 고릅니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

단일 규칙만 쓰면 너무 엄격하거나 너무 느슨해질 수 있습니다. ToT는 중간 균형점을 찾는 데 도움이 됩니다.

### 멘토에게 설명할 말

```text
메일에서 뽑은 항목을 무조건 필수로 강제하면 오탐이 생길 수 있어,
Strict/Balanced/Loose 후보를 만들고 실제 파일 컬럼과 비교해 규칙을 선택했습니다.
```

## 18. Self-Correction

### 무엇인가요?

Self-Correction은 검증 후 일부 오류를 스스로 고쳐보고 다시 검사하는 방식입니다.

초등학생식으로 말하면, Self-Correction은 "틀린 답을 보고 고칠 수 있는 것만 고친 다음 다시 채점하는 과정"입니다.

### 왜 썼나요?

모든 오류를 다시 제출하게 하면 사용자의 부담이 큽니다. 하지만 모든 오류를 자동 수정하면 위험합니다. 그래서 안전하게 고칠 수 있는 것만 자동 교정했습니다.

자동 교정 가능한 예시는 다음입니다.

- `2026/06/05` -> `2026-06-05`
- `2026.06.05` -> `2026-06-05`
- `매우 급함` -> `상`
- `긴급` -> `상`

자동 교정하지 않는 예시는 다음입니다.

- 담당자 이름 누락
- 부서명 누락
- 중복 데이터 중 어떤 행을 살릴지 결정

### 누가 교정값을 정하나요? (LLM 제안 → 코드 검증)

이 부분이 이번 과제의 **에이전틱 핵심**입니다. 교정값을 정하는 방식이 두 갈래입니다.

1. **PROPOSE (🧠 LLM 제안)**: Azure OpenAI가 각 오류의 교정값을 근거와 함께 제안합니다.
   예: 긴급도 `매우 급함` → 제안 `상`, 근거 "‘매우 급함’은 ‘상’과 의미가 가장 유사함".
   키워드 사전에 없는 표현(`몹시 시급`, `급을 다툼` 등)도 의미로 판단할 수 있습니다.
2. **VERIFY (⚙️ 코드 검증)**: LLM 제안값을 **허용값 목록·날짜형식으로 재검증**합니다.
   `상/중/하`에 없는 값이거나 실제 날짜로 파싱되지 않으면 **버립니다**(환각 차단).
   LLM 제안이 게이트를 통과하지 못하면 결정론 키워드 매핑으로 1회 폴백합니다.
3. **RE-VALIDATE (⚙️ 코드 확정)**: 교정본을 다시 검증해 **오류 수가 줄 때만** 채택합니다.

즉 LLM은 "이 자연어 값이 무슨 뜻인지"를 판단하고, 코드는 "그 제안이 규칙에 맞는지"를 확정합니다.
Azure 키가 없으면 전체가 결정론 키워드·날짜 정규화로 폴백하므로, **오프라인에서도 재현 가능**합니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

아예 자동 교정을 하지 않으면 사용자가 고쳐야 할 일이 많아집니다. 반대로 LLM에게 교정을 전부 맡겨
바로 확정하면 없는 값을 지어낼 위험이 있습니다. 그래서 **LLM은 제안까지, 확정은 결정론 게이트**로
분리했습니다. 필수값 누락·중복은 데이터를 지어내야 하므로 애초에 제안 대상에서 제외합니다.

### 멘토에게 설명할 말

```text
Self-Correction은 LLM이 교정값을 '제안'하고 코드가 허용값·재검증으로 '확정'하는
Propose–Verify 루프입니다. LLM의 자연어 판단력은 쓰되, 환각·비결정성은 결정론 게이트가 막습니다.
필수값 누락·중복은 업무 판단이 필요하므로 재제출 대상으로 남깁니다.
Azure 키가 없으면 결정론 규칙으로 폴백해 오프라인에서도 같은 결과가 나옵니다.
```

## 19. 규칙 기반 검증

### 무엇인가요?

규칙 기반 검증은 AI가 감으로 판단하는 것이 아니라, 코드로 정해진 규칙에 따라 검사하는 방식입니다.

초등학생식으로 말하면, 규칙 기반 검증은 "정답지가 있는 채점"입니다.

이 프로젝트의 검증 규칙은 네 가지입니다.

- 필수값 누락
- 날짜 형식 오류
- 허용되지 않은 코드값
- 중복 데이터

### 왜 썼나요?

엑셀 검증은 창의적인 답보다 정확한 답이 중요합니다. 같은 입력이면 항상 같은 오류 보고서가 나와야 합니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

LLM에게 엑셀 검증을 맡기면 설명은 잘하지만, 행 번호나 오류 유형이 매번 완전히 같다는 보장이 약합니다. 그래서 검증과 병합은 결정론적 코드로 처리했습니다.

### 멘토에게 설명할 말

```text
엑셀 검증은 LLM이 아니라 규칙 기반 코드로 처리했습니다.
그래야 같은 입력에서 같은 결과가 나오고, 오류 보고서의 신뢰성을 확보할 수 있습니다.
```

## 20. RAG 구조

### 무엇인가요?

RAG는 AI가 모르는 내용을 외부 문서에서 찾아 참고하는 방식입니다. 예를 들어 컬럼 정의서나 코드값 기준 문서를 찾아볼 수 있습니다.

초등학생식으로 말하면, RAG는 "AI가 문제를 풀기 전에 참고서를 찾아보는 방식"입니다.

### 실제로 무엇에 쓰고 있나요? (내 발송 스타일 RAG)

이번 구현에서 RAG는 **실제로 동작하는 핵심 기능**입니다. 요청 메일 초안을 만들 때, **내가 과거에 보냈던 요청 메일들을 참고 문서처럼 검색해서 프롬프트에 톤 예시로 넣습니다.**

```text
① 내가 쓴 과거 메일 업로드(.txt/.md/.eml) 또는 붙여넣기
   → docs/reference/style_samples/ 에 저장 (스타일 코퍼스)
② 요청 메일 초안 생성 시 retrieve_style_samples()로 관련 과거 메일 검색
③ create_request_mail() 프롬프트에 "이 인사말·톤·구성을 따라 써라"로 주입
④ 결과 초안에 "내 발송 스타일 반영됨" 배지 표시
```

중요: 이건 **모델을 다시 학습시키는 게 아닙니다.** 과거 메일을 검색해서 프롬프트에 참고로 넣는 검색증강(RAG) 방식입니다.

### 검색 방식과 후속 확장

현재 검색은 `docs/reference` 폴더 대상 **키워드 기반 검색**입니다(FAISS/임베딩 아님). 회사 컬럼 정의서·코드값 기준·과거 취합 사례를 붙이는 본격 Vector DB는 후속 확장으로 남겼습니다.

### 멘토에게 설명할 말

```text
RAG를 두 가지로 씁니다.
(1) 실제 사용: 내 과거 발송 메일을 검색해 요청 메일 초안 톤에 반영 — 학습이 아니라 검색 주입입니다.
(2) 후속 확장: 컬럼 정의서·코드값 기준 문서 검색은 FAISS 기반으로 확장 가능하게 인터페이스만 고정했습니다.
엑셀 검증에는 RAG를 쓰지 않습니다. 검증은 재현 가능한 규칙이 필요하기 때문입니다.
```

## 21. FAISS

### 무엇인가요?

FAISS는 문서를 숫자 벡터로 바꿔서 비슷한 문서를 빠르게 찾는 검색 도구입니다.

초등학생식으로 말하면, FAISS는 "비슷한 내용을 가진 참고서를 빨리 찾아주는 도서관 검색기"입니다.

### 현재 프로젝트에서의 상태

현재는 운영 확장 후보입니다. 1차 PoC에서는 본격적으로 사용하지 않았습니다.

### 왜 지금 본격 사용하지 않았나요?

현재 과제의 핵심은 엑셀 검증/병합 자동화의 가치를 증명하는 것입니다. FAISS 기반 RAG까지 완성하려면 문서 수집, 임베딩, 권한 관리, 검색 품질 평가가 추가로 필요합니다. 그래서 후속 확장 범위로 분리했습니다.

### 멘토에게 설명할 말

```text
FAISS는 기준 문서 검색을 위한 후속 확장 후보입니다.
이번 PoC에서는 핵심 검증 기능에 집중하기 위해 본격 운영하지 않았습니다.
```

## 22. Langfuse

### 무엇인가요?

Langfuse는 LLM 호출과 Agent 실행 과정을 추적하는 모니터링 도구입니다.

초등학생식으로 말하면, Langfuse는 "AI가 어떤 생각 순서로 일했는지 기록하는 관찰 노트"입니다.

### 현재 프로젝트에서의 상태 (연동 완료)

이번에 **실제로 연동을 완료했습니다.** `langfuse` v2 패키지를 설치하고, US 리전 프로젝트에 키를 연결했으며, `USE_LANGFUSE=true`로 켰습니다. `auth_check` 통과와 테스트 트레이스 전송을 확인했습니다.

- 파이프라인의 **모든 노드**(요구분석·계획·검증·자기수정·병합·리포트)에서 `trace_execution`이 호출되어, 요청 1건마다 실행 순서·입력·출력 요약·오류 레벨이 Langfuse 대시보드에 기록됩니다.
- 결과물(검증/병합 엑셀)은 바뀌지 않습니다. Langfuse는 **관측(observability)** 만 추가합니다.

### 왜 넣었나요?

시연 가이드의 핵심 원칙이 "결과보다 과정(추론 로그)"입니다. Langfuse 대시보드로 **멀티에이전트 실행 과정을 시각적으로** 보여줄 수 있어, 각 에이전트가 어떤 입력을 받아 무엇을 했는지 추적 가능함을 증명합니다.

### 멘토에게 설명할 말

```text
Langfuse를 실제로 연동해 각 에이전트 노드 실행을 대시보드에서 추적할 수 있습니다.
코드가 v2 API(client.trace/span)를 쓰므로 langfuse<3으로 버전을 고정했고, 키 리전(US)을 맞춰 연결했습니다.
민감 원문은 남기지 않고 request_id·단계·요약 중심으로 기록합니다.
```

## 23. Gmail API

### 무엇인가요?

Gmail API는 코드에서 Gmail을 통해 이메일을 보낼 수 있게 해주는 API입니다.

초등학생식으로 말하면, Gmail API는 "프로그램이 Gmail 대신 편지를 보낼 수 있게 해주는 우체통 열쇠"입니다.

### 발송 기능 (요청 메일 + 리마인드 + 첨부)

요청 메일 초안과 리마인드 메일을 **편집 가능한 폼**(제목/본문/첨부/다중 수신자)으로 만든 뒤 `POST /api/send-request-mail`로 발송합니다. 요청과 함께 받은 **양식 엑셀을 그대로 첨부**해 보낼 수 있습니다(multipart).

### 왜 기본값은 Mock인가요?

실제 이메일 발송은 위험합니다. 잘못 누르면 실제 사람에게 메일이 갑니다. 그래서 기본은 Mock, 즉 보낸 척만 하는 안전 모드입니다. 실제 Gmail 발송은 `EMAIL_SEND_MODE=gmail`과 OAuth 인증이 있을 때만 가능하도록 분리했습니다.

### Gmail API vs Gmail MCP (중요한 구분)

메일을 **읽어오는** 부분은 앱이 하지 않습니다. Claude Code의 **Gmail MCP**로 필요할 때만 가져옵니다.

```text
Gmail API (앱) = FastAPI 백엔드가 OAuth로 직접 메일을 발송하는 기능
Gmail MCP (에이전트) = Claude Code가 도구로 Gmail을 읽어 팀장 포워딩 메일·과거 발송 메일을 가져오는 것
```

즉 "앱이 수신함을 상시 자동 감시"하는 게 아니라, 사용자가 필요할 때 Claude Code에게 요청하면 MCP로 메일을 가져와 앱 워크플로우에 넣는 **Human-in-the-loop** 방식입니다.

### 멘토에게 설명할 말

```text
메일 발송은 실제 영향이 있으므로 기본값은 Mock, 실제 발송은 OAuth 설정 시에만 동작합니다.
메일 읽기는 앱이 직접 하지 않고 Claude Code의 Gmail MCP로 필요할 때만 가져오는 방식으로 설계했습니다.
앱 백엔드의 자동 수신함 검색은 후속 확장 범위입니다.
```

## 24. Local File Storage

### 무엇인가요?

Local File Storage는 결과 파일을 로컬 폴더에 저장하는 방식입니다.

초등학생식으로 말하면, Local File Storage는 "작업 결과를 내 컴퓨터 폴더에 저장하는 방식"입니다.

이 프로젝트에서는 다음 폴더를 사용합니다.

- `data/error_reports`

### 다른 것을 쓰는 건 왜 덜 맞았나요?

DB, AWS S3, 사내 파일 저장소도 가능하지만 1차 PoC에는 과합니다. 먼저 기능 가치와 검증 정확도를 보여주는 것이 중요했습니다.

### 멘토에게 설명할 말

```text
1차 PoC에서는 결과 파일을 쉽게 확인할 수 있도록 로컬 파일 저장 방식을 사용했습니다.
운영 단계에서는 DB나 사내 저장소로 확장할 수 있습니다.
```

### 멘토에게 설명할 말


## 27. CORS

### 무엇인가요?

CORS는 서로 다른 주소에서 실행되는 프론트엔드와 백엔드가 통신할 수 있게 허용하는 설정입니다.

초등학생식으로 말하면, CORS는 "다른 교실에 있는 친구가 말을 걸어도 되는지 허락하는 규칙"입니다.

### 왜 썼나요?

개발 중 React는 보통 `localhost:5173`, FastAPI는 `127.0.0.1:8000`에서 실행됩니다. 주소가 다르기 때문에 API 요청을 허용하는 설정이 필요합니다.

### 다른 것을 쓰는 건 왜 덜 맞았나요?

프록시만으로도 일부 해결할 수 있지만, API 서버 자체에서 CORS를 명확히 허용하는 것이 개발 중에는 편합니다. 운영에서는 특정 도메인만 허용해야 합니다.

### 멘토에게 설명할 말

```text
로컬 개발에서 React와 FastAPI가 서로 다른 포트에서 실행되므로 CORS 설정을 적용했습니다.
운영에서는 허용 도메인을 제한해야 합니다.
```

```text
현재 56개 테스트가 통과합니다(검증·병합·ToT·Self-Correction·가이드·제출추적·스타일RAG·프로젝트 공통항목 동기화·메일발송·API 포함).
```

### 멘토에게 설명할 말

```text
기술 구현뿐 아니라 문제 정의, 아키텍처, 검증 결과를 발표용으로 전달하기 위해 PPT 산출물을 만들었습니다.
```

## 32. 왜 LLM으로 모든 것을 하지 않았나요?

멘토가 가장 많이 물어볼 수 있는 질문입니다.

### 답변

LLM은 메일처럼 사람 말로 된 내용을 이해하는 데 강합니다. 하지만 엑셀 검증은 창의적인 답이 아니라 정확한 답이 필요합니다.

예를 들어 엑셀 검증에서는 다음이 중요합니다.

- 몇 번째 행이 오류인지
- 어떤 컬럼이 오류인지
- 오류 유형이 무엇인지
- 같은 입력에서 같은 결과가 나오는지

LLM은 설명을 잘하지만, 같은 입력에서 항상 같은 오류 보고서를 만든다는 보장이 약합니다. 그래서 LLM은 메일 분석에만 사용하고, 검증은 규칙 기반 코드로 분리했습니다.

### 멘토에게 말할 문장

```text
AI의 유연함은 메일 이해에 사용하고,
업무 데이터 검증은 결정론적 코드로 처리해 신뢰성과 재현성을 확보했습니다.
```

## 33. 왜 LangGraph가 필요했나요?

### 답변

단순히 Python 함수만 순서대로 호출해도 동작은 합니다. 하지만 이 과제의 목표는 단순 자동화가 아니라 멀티 에이전트 구조를 보여주는 것입니다.

LangGraph를 쓰면 다음을 명확히 보여줄 수 있습니다.

- Requirement Analysis Agent가 메일을 분석한다.
- Supervisor Agent가 계획과 검증 규칙을 선택한다.
- Excel Validation Agent가 파일을 검증한다.
- Self-Correction Agent가 안전한 오류만 교정한다.
- Report Agent가 결과를 요약한다.

### 멘토에게 말할 문장

```text
LangGraph를 사용해 Agent별 역할과 실행 순서를 명확하게 분리했습니다.
그래서 단순 Tool Call이 아니라 상태 기반 Multi-Agent 워크플로우임을 보여줄 수 있습니다.
```

## 34. Gmail 연동은 어떻게 했나요?

### 답변

메일 **읽기**와 **발송**을 분리해서 설계했습니다.

- **읽기**: 앱이 수신함을 상시 자동 감시하지 않습니다. 대신 Claude Code의 **Gmail MCP**로 필요할 때만 가져옵니다(팀장 포워딩 메일, 내 과거 발송 메일). 사용자가 요청하면 MCP가 메일을 읽어 앱 워크플로우에 넣는 Human-in-the-loop 방식입니다.
- **발송**: 요청/리마인드 메일을 폼으로 만들어 발송합니다. 기본값은 Mock이고, `EMAIL_SEND_MODE=gmail` + OAuth일 때만 실제 Gmail 발송됩니다.

앱 백엔드가 OAuth로 수신함을 직접 검색·자동 수집하는 것은 후속 확장 범위입니다.

### 멘토에게 말할 문장

```text
메일 읽기는 앱이 직접 하지 않고 Claude Code의 Gmail MCP로 필요할 때만 가져오는 Human-in-the-loop로 설계했습니다.
발송은 기본 Mock, 실제 발송은 OAuth 설정 시에만 동작하도록 분리했습니다.
앱 자체의 수신함 자동 검색은 후속 확장입니다.
```

## 35. 왜 DB를 쓰지 않았나요?

### 답변

현재는 로컬 PoC입니다. 사용자가 결과 파일을 바로 확인하는 것이 중요합니다. 그래서 병합 파일과 오류 보고서를 로컬 폴더에 생성했습니다.

운영 단계에서는 DB, 사내 파일 저장소, S3 같은 저장소로 확장할 수 있습니다.

### 멘토에게 말할 문장

```text
1차 PoC에서는 결과 파일을 바로 확인하는 것이 중요해 Local File Storage를 사용했습니다.
운영 전환 시 DB나 사내 저장소로 확장할 수 있습니다.
```

## 37. 예상 질문과 답변

### Q1. 왜 LLM에게 엑셀 검증까지 맡기지 않았나요?

엑셀 검증은 정확성과 재현성이 중요하기 때문입니다. LLM은 자연어 설명에는 강하지만, 행 번호와 오류 유형을 항상 동일하게 보장하기 어렵습니다. 그래서 검증은 pandas/openpyxl 기반 규칙으로 처리했습니다.

### Q2. LangGraph 없이도 구현 가능하지 않나요?

가능합니다. 하지만 이 과제는 멀티 에이전트 흐름을 보여주는 것이 중요했습니다. LangGraph를 쓰면 각 Agent의 역할과 실행 순서를 명확하게 표현할 수 있습니다.

### Q3. ToT는 실제로 어디에 쓰였나요?

검증 규칙 선택에 사용했습니다. Strict, Balanced, Loose 후보를 만들고 실제 업로드 파일의 컬럼과 비교해 과잉 필수 규칙 오탐을 줄였습니다.

### Q4. Self-Correction은 위험하지 않나요?

그래서 범위를 제한했습니다. 날짜 형식과 코드값처럼 의미가 명확한 것만 자동 교정하고, 필수값 누락과 중복은 재제출 대상으로 남겼습니다.

### Q5. 실제 Gmail 연동은 됐나요?

읽기는 Claude Code의 Gmail MCP로 필요할 때 가져오는 방식(Human-in-the-loop)으로 동작합니다. 발송은 기본 Mock이고, OAuth 설정 시 실제 Gmail 발송이 가능합니다. 앱 백엔드가 수신함을 자동 검색·수집하는 것은 후속 확장 범위입니다.

### Q6. RAG는 구현된 건가요?

네, 실제로 씁니다. **내가 과거에 보낸 요청 메일을 검색해서 요청 메일 초안의 톤·구성에 반영하는 스타일 RAG**를 구현했습니다(모델 학습이 아니라 검색 주입). 현재는 `docs/reference/style_samples` 대상 키워드 검색이며, 컬럼 정의서·코드값 기준 문서를 붙이는 FAISS 기반 Vector DB는 후속 확장입니다. 엑셀 검증에는 RAG를 쓰지 않습니다.

### Q7. 성능은 어느 정도인가요?

테스트 기준으로 4개 파일, 46행 데이터셋에서 오류 검출 F1 100%, 재현성 표준편차 0, 자동 처리 약 0.05초대 수준을 확인했습니다.

### Q8. 원본 엑셀 파일을 수정하나요?

아니요. 원본 파일은 보존하고, 병합 파일과 오류 보고서를 별도 경로에 생성합니다.

### Q9. 운영에 바로 쓸 수 있나요?

현재는 1차 PoC입니다. 운영 전환을 위해서는 인증, 권한 관리, 개인정보 마스킹, 사내 메일/문서 시스템 연동, 감사 로그가 추가로 필요합니다.

### Q10. 이 프로젝트의 핵심 차별점은 무엇인가요?

LLM을 무조건 많이 쓴 것이 아니라, LLM이 필요한 부분과 필요 없는 부분을 나누었습니다. 자연어 이해는 LLM에게 맡기고, 업무 데이터 검증은 결정론적 코드로 처리해 신뢰성을 높인 점이 핵심입니다.

### Q11. "내 스타일로 요청 메일 작성"은 어떻게 동작하나요?

내가 과거에 보낸 요청 메일을 업로드(.txt/.md/.eml)하거나 붙여넣어 저장하면, 요청 초안 생성 시 그 메일들을 검색해 "이 인사말·톤·구성을 따라 써라"로 프롬프트에 넣습니다. 결과 초안에 "내 발송 스타일 반영됨" 배지가 표시됩니다. 모델을 학습시키는 게 아니라 검색해서 참고로 넣는 RAG 방식입니다.

### Q12. Langfuse는 실제로 연동됐나요?

네. langfuse v2를 설치하고 US 리전 키를 연결해 `USE_LANGFUSE=true`로 켰습니다. 파이프라인의 모든 에이전트 노드 실행이 대시보드에 기록되어, 멀티에이전트가 어떤 순서로 어떤 입력·출력을 처리했는지 시각적으로 추적할 수 있습니다.

### Q13. 화면(UI)은 어떻게 구성돼 있나요?

취합 업무 순서 그대로 4단계(① 요청 메일 보내기 ② 검증·병합 ③ 제출 현황·리마인드 ④ 공통 항목 수정)로 구성했습니다. 각 단계는 왼쪽 컨트롤, 오른쪽 결과/발송 폼으로 나뉘고, 요청 메일과 리마인드 메일은 동일한 발송 폼(제목·본문·첨부·다중 수신자)을 씁니다. 02 검증 결과에는 **‘에이전트 추론 타임라인’ 패널**이 있어 각 단계의 판단 주체(🧠 LLM / ⚙️ 규칙)와 판단 내용을 보여주고, **추론 트레이스(.md/.json)를 다운로드**할 수 있습니다.

### Q14. "개별 기능 4개"가 아니라 왜 하나의 에이전트라고 하나요?

02 검증을 실행하면 내부적으로 ANALYZE(🧠LLM) → PLAN(🧠LLM) → BRANCH×3/SELECT(⚙️규칙) → VALIDATE(⚙️규칙) → CRITIQUE → PROPOSE(🧠LLM)/VERIFY(⚙️규칙) → RE-VALIDATE → MERGE/REPORT가 **하나의 상태(AgentState)를 공유하며 이어집니다**. 화면의 01~04는 사람이 개입하는 지점(Human-in-the-loop)일 뿐, 검증 단계 자체는 여러 에이전트가 한 세션으로 협업합니다. 그 증거로 요청마다 추론 타임라인과 트레이스 파일이 남습니다.

### Q15. LLM을 능동적으로 쓴 곳이 어디인가요? (Claude 웹 요약과의 차이)

두 곳에서 LLM이 **판단**합니다. ① Supervisor가 실제 업로드 컬럼을 보고 검증 전략·리스크를 계획합니다. ② Self-Correction에서 LLM이 애매한 값의 교정을 근거와 함께 제안합니다. 둘 다 코드가 **검증**합니다 — ToT 규칙은 coverage 게이트로, 교정 제안은 허용값·재검증 게이트로 확정합니다. 단발 요약과 달리 LLM이 워크플로우의 판단을 담당하되, 정확성이 필요한 확정은 결정론 코드가 책임집니다.

### Q16. 각 단계 로그는 어떻게 증명하나요?

요청 1건이 끝나면 `data/traces/{요청ID}.json`(구조화 원본)과 `.md`(사람이 읽는 타임라인 표)가 자동 저장됩니다. 표에는 순서·에이전트·국면·**판단 주체(LLM/규칙)**·판단 내용이 들어가고, 자가교정 상세에는 `원본 → 교정`, 주체, **LLM 근거**가 기록됩니다. 시연 영상에서 이 파일을 그대로 보여주면 "과정을 기록·증명한다"는 요구를 충족합니다.

### 실제 업무 흐름 요약 (멘토 설명용)

```text
팀장이 취합 메일을 나에게 포워딩
  → (Claude Code + Gmail MCP) 그 메일과 내 과거 발송 메일을 가져옴
  → 앱이 요구사항 분석 + 내 스타일 RAG로 요청 메일 초안 생성
  → 양식 엑셀 첨부해 제출자에게 발송
  → 제출자가 엑셀을 작성해 회신
  → 앱이 결정론적 규칙으로 검증·병합, 오류 보고서·취합 엑셀 생성
  → 미제출자에게 리마인드 발송
```
